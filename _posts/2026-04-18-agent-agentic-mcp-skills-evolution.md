요즘 AI 관련 글을 읽으면 용어 폭탄이 날아온다. **에이전트(Agent)**, **Agentic AI**, **MCP**, **Skills**... 각각 따로 설명한 글은 많은데, 이것들이 왜 같이 언급되는지, 어떤 맥락에서 나온 건지 설명해주는 글은 의외로 드물다.

결론부터 말하면, 이 네 가지 개념은 **하나의 진화 흐름** 위에 있다. AI가 "답하는 기계"에서 "일하는 기계"로 바뀌는 과정에서, 각 단계의 핵심 문제를 해결하기 위해 순서대로 등장했다. 한 줄기로 이어서 보면 전부 이해된다.

---

## 출발점: AI는 원래 질문-답변 기계였다

ChatGPT 초창기를 떠올려보자. 우리는 질문을 입력했고, AI는 텍스트로 답했다. 그게 전부였다.

이 구조의 한계는 명확하다. AI가 아무리 똑똑한 답을 내놓아도, **실제로 아무것도 하지 않는다**. "오늘 날씨 어때?"라고 물으면 날씨 API를 호출하는 게 아니라, 학습 데이터에서 날씨에 대한 지식을 꺼내 답할 뿐이다. "이 파일 분석해줘"라고 하면 파일을 읽는 게 아니라, 내가 파일 내용을 복사해서 붙여넣어야 했다.

**AI가 "생각"만 하고 "행동"은 못 하는 시대**였다.

---

## 1단계: Agent — AI에게 손발을 달아주다

이 문제를 해결한 개념이 **AI 에이전트(Agent)**다.

에이전트는 LLM(대형 언어 모델)에 **도구(Tool)**를 붙인 구조다. 단순하다. AI가 "지금 날씨 API를 호출해야겠다"고 판단하면, 실제로 그 API를 호출하고 결과를 받아서 답한다. 파일을 읽어야 하면 파일 시스템에 접근한다. 코드를 실행해야 하면 실행 환경을 호출한다.

핵심 변화는 LLM이 **텍스트만 생성하는 게 아니라, 행동을 결정하는 두뇌**가 됐다는 점이다. 실제 작업은 도구가 수행하고, LLM은 "언제, 어떤 도구를, 어떤 파라미터로 호출할지"를 결정한다.

```
사용자 요청
     ↓
 LLM (두뇌)
     ↓  어떤 도구를 쓸까?
 Tool 호출 (손발)
     ↓  결과 반환
 LLM이 결과 해석
     ↓
  최종 응답
```

에이전트가 가능해지면서, AI는 처음으로 **실세계와 연결**됐다. 웹 검색, 파일 읽기/쓰기, 코드 실행, 외부 API 호출... 이제 AI는 말만 하지 않고 실제로 무언가를 한다.

---

## 2단계: Agentic AI — 한 번이 아니라 루프로 돌린다

에이전트가 등장했지만, 초기엔 한계가 있었다. 도구를 한 번 쓰고 끝나는 경우가 많았다. 복잡한 작업은 여전히 사람이 중간중간 개입해서 다음 단계를 지시해야 했다.

**Agentic AI**는 이 구조를 한 단계 더 밀어붙인다. 목표를 한 번 주면, AI가 **스스로 계획을 세우고, 실행하고, 결과를 보고, 다음 단계를 결정하는 루프**를 계속 돌린다. 중간에 실패하면 원인을 분석해서 재시도한다. 목표가 달성되거나 불가능하다고 판단될 때까지.

이 루프를 **ReAct 패턴**이라고도 부른다. Reasoning(추론) + Acting(행동)의 반복.

```
목표 설정
   ↓
[추론] 다음에 뭘 해야 할까?
   ↓
[행동] 도구 호출
   ↓
[관찰] 결과 확인
   ↓
[판단] 목표 달성됐나? → 아니면 다시 추론으로
   ↓
완료
```

"PR 초안 작성해줘"라고 하면, Agentic AI는 코드 변경사항을 읽고, 관련 테스트를 확인하고, 이전 커밋 메시지를 참고해서 PR 설명을 쓰고, 리뷰어 추천까지 한다. 이 모든 걸 사람의 개입 없이.

**Agentic AI의 핵심은 자율성이다.** 도구를 쓸 수 있느냐(Agent)를 넘어서, 얼마나 자율적으로 다단계 작업을 완수하느냐(Agentic)의 문제다.

---

## 3단계: MCP — 도구를 표준화하다

Agentic AI가 가능해지면서 새로운 문제가 생겼다. 에이전트마다 도구를 연결하는 방식이 제각각이었다.

- Claude는 자체 Tool Use API 방식
- OpenAI는 Function Calling 방식
- LangChain은 또 다른 추상화 레이어
- 각 서비스(GitHub, Notion, Slack 등)는 연동하려는 AI마다 따로 만들어야

개발자 입장에서는 악몽이다. GitHub MCP를 만들면 Claude에서 쓸 수 있는데, GPT-4에서 쓰려면 다시 만들어야 한다. AI 클라이언트가 10개면 같은 도구를 10번 만드는 셈이다.

**MCP(Model Context Protocol)**는 Anthropic이 2024년에 공개한 오픈 표준으로, 이 파편화 문제를 해결하기 위해 나왔다. AI 세계의 USB-C다.

핵심 구조는 세 가지다:

- **MCP Host**: AI 클라이언트 앱 (Claude Desktop, Cursor, VS Code 등)
- **MCP Client**: Host 내에서 서버와 통신하는 레이어
- **MCP Server**: 실제 기능을 제공하는 서버 (GitHub, Notion, 파일시스템 등)

```
[Claude Desktop]  [Cursor]  [VS Code]
       ↓              ↓          ↓
  MCP Client    MCP Client   MCP Client
       ↓              ↓          ↓
        ↘            ↓          ↙
         [MCP Protocol (표준)]
                  ↓
        [MCP Server: GitHub]
        [MCP Server: Notion]
        [MCP Server: 파일시스템]
```

MCP 서버를 한 번 만들면, MCP를 지원하는 모든 AI 클라이언트에서 쓸 수 있다. Anthropic이 공개한 이후 OpenAI, Google, MS 등 주요 플레이어들이 빠르게 지원을 선언하면서, 사실상 **업계 표준**으로 자리잡아가고 있다.

MCP가 해결한 건 단순히 "연결의 표준화"가 아니다. AI 생태계가 **재사용 가능한 도구 레이어**를 갖게 됐다는 것이다. 도구를 한 번 만들면 어디서든 쓸 수 있고, 에이전트는 표준 방식으로 세상의 모든 도구와 연결된다.

---

## 4단계: Skills — 도구를 넘어, 능력을 조합하다

MCP로 도구 문제가 해결되자, 다음 질문이 생겼다. 도구는 원자적(atomic)이다. "파일 읽기", "API 호출", "검색하기"... 각각 단순한 동작이다.

그런데 실제 업무는 이런 단순 동작의 조합으로 이루어진다. "PR 리뷰하기"는 코드 읽기 + 이슈 확인 + 컨텍스트 파악 + 리뷰 작성의 조합이다. "버그 수정하기"는 에러 로그 분석 + 관련 코드 탐색 + 수정 + 테스트 실행의 조합이다.

**Skills**는 이런 복합적인 능력을 패키징한 개념이다. 단순 도구 호출이 아니라, 특정 목표를 달성하기 위한 **재사용 가능한 전문 능력 단위**다.

Claude Code에서 `/commit`, `/review-pr` 같은 슬래시 커맨드가 이 개념의 구현체다. 커밋 생성이라는 목표를 위해 "변경사항 분석 → 메시지 초안 → 스테이징 → 커밋 실행"의 전체 흐름이 패키징되어 있다. 사용자는 `/commit`만 입력하면 된다.

Skill의 세 가지 특성:

**재사용성** — 한 번 정의하면 반복해서 쓸 수 있다. 매번 "파일 읽고, 분석하고, 요약해줘"를 장황하게 지시하지 않아도 된다.

**조합성** — Skill은 다른 Skill을 호출할 수 있다. 복잡한 워크플로우를 레고 블록처럼 조립한다. Multi-Agent 시스템에서 오케스트레이터 에이전트가 여러 서브 에이전트(각각의 Skill)에 작업을 위임하는 구조가 여기서 나온다.

**맥락 인식** — 단순 템플릿이 아니다. Skill은 현재 상황(코드베이스, 이슈 상태, 팀 컨벤션 등)을 인식하고 그에 맞게 실행된다.

## 전체 그림: 하나의 스택으로 보기

이제 네 개념이 어떻게 쌓이는지 보인다.

```
┌─────────────────────────────────────┐
│           Skills (능력 레이어)        │
│  재사용 가능한 전문 능력 단위           │
│  /commit, /review-pr, 커스텀 워크플로우│
├─────────────────────────────────────┤
│           MCP (연결 레이어)           │
│  표준화된 도구 프로토콜                 │
│  GitHub, Notion, DB, 파일시스템...    │
├─────────────────────────────────────┤
│        Agentic AI (실행 레이어)       │
│  자율적 루프 실행, 계획-행동-관찰       │
├─────────────────────────────────────┤
│          Agent (기반 레이어)          │
│  LLM + 도구 = 행동 가능한 AI           │
└─────────────────────────────────────┘
```

- **Agent**가 AI에게 손발을 달아줬다면
- **Agentic AI**는 그 손발로 혼자 일을 끝내는 능력을 줬고
- **MCP**는 연결할 수 있는 도구의 범위를 표준화해서 무한히 넓혔으며
- **Skills**는 이 전체를 재사용 가능한 능력 단위로 패키징했다

각 계층은 아래 계층 위에 쌓이면서 새로운 문제를 해결한다. 독립적인 개념이 아니라 **진화의 산물**이다.

---

## 지금 어디쯤인가

2026년 현재, 이 스택은 빠르게 성숙하고 있다.

MCP는 업계 표준으로 굳어지는 중이다. 커뮤니티에서 만든 MCP 서버가 수천 개를 넘었고, 주요 SaaS들이 공식 MCP 서버를 제공하기 시작했다. Claude, Cursor, VS Code, Zed 같은 툴들이 MCP를 기본 지원한다.

Skills 레이어는 아직 표준화가 진행 중이다. Claude Code의 슬래시 커맨드, LangChain의 체인/에이전트, AutoGPT의 플러그인 시스템... 각자의 방식으로 구현하고 있다. 이 레이어의 표준화가 다음 과제다.

남은 과제는 **신뢰와 안전**이다. Agentic AI가 자율적으로 일을 할수록, "어디까지 허용할 것인가"의 문제가 커진다. 파일을 삭제해도 되나? 외부 API에 데이터를 보내도 되나? 이 문제를 얼마나 잘 풀어내느냐가 다음 단계의 핵심이다.

---

지금 이 글을 읽고 있다면, 이미 이 흐름의 한가운데 있는 것이다. AI 도구들이 빠르게 이 스택을 채워나가고 있다. 개념을 알고 쓰면, 다음에 새로운 무언가가 나왔을 때 "이게 스택의 어디에 해당하는 건가"를 빠르게 파악할 수 있다. 그게 결국 더 빨리 활용하는 방법이다.
