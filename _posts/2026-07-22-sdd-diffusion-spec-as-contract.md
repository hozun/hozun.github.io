지난 4월에 [spec-kit으로 SDD 입문](/2026/04/20/spec-kit-spec-driven-development.html) 글을 썼다. 그때는 "이런 방법론과 툴킷이 있다" 수준의 소개였다. 3개월이 지난 지금, 상황이 꽤 달라졌다.

결론부터 말하면 이렇다.

- **SDD는 "실험"에서 "팀 표준"으로 넘어가는 구간에 있다.** 아직 업계 전체의 기본값은 아니지만, AI-네이티브 개발에 진지한 조직에서는 디폴트가 되어가는 중이다.
- **도구 지형이 3파전으로 정리됐다.** spec-kit(오픈소스 레퍼런스), Kiro(AWS의 IDE), Tessl(spec-as-source 극단).
- **가장 중요한 변화는 스펙이 "문서"에서 "계약(contract)"이 됐다는 것.** 도메인 전문가나 기획이 쓴 스펙을 AI가 받아 실제 프로덕션을 만드는 사례가 나오기 시작했다.

세 번째가 이 글의 핵심이고, 사용자가 궁금해한 "스펙을 받아서 개발되는 사례"에 해당한다.

---

## 4월 이후 무엇이 달라졌나

지난 글에서는 spec-kit 하나만 이야기했다. 지금은 도구가 각자 다른 철학으로 갈라졌다.

| 도구 | 만든 곳 | 성격 | 특징 |
|---|---|---|---|
| **Spec Kit** | GitHub | 오픈소스 CLI | 스타 9.3만+, v0.8.7(5월), 30개+ 에이전트 호환. 사실상 레퍼런스 구현 |
| **Kiro** | AWS | Agentic IDE | Code OSS 기반, EARS 표기법, Bedrock 라우팅. "스펙 = 작업 단위" |
| **Tessl** | Tessl | spec-as-source 플랫폼 | 코드에 `// GENERATED FROM SPEC - DO NOT EDIT`, `tessl build`로 재생성 |

여기에 Claude Code의 `/sdd:specify` 계열 커맨드, Cursor의 Plan Mode, OpenSpec, BMAD-METHOD 등이 각 에이전트 진영에서 SDD 워크플로우를 흡수했다. 4월에는 "spec-kit을 설치한다"였다면, 지금은 **이미 쓰는 에이전트 안에 SDD 흐름이 들어와 있다**는 게 차이다.

특히 눈에 띄는 건 Kiro가 항공우주 요구공학에서 쓰던 **EARS 표기법**을 끌어온 점이다.

```
# EARS (Easy Approach to Requirements Syntax) 예시

WHEN 사용자가 만료된 매직링크를 클릭하면,
  시스템은 SHALL 재발송 안내 화면을 보여준다.

WHILE 결제가 처리 중인 동안,
  시스템은 SHALL 중복 제출 버튼을 비활성화한다.
```

"로그인 잘 되게 해줘" 같은 모호한 문장 대신, **트리거-응답이 명시된 검증 가능한 문장**으로 요구사항을 적는다. 이게 뒤에 나올 "스펙이 계약이 된다"의 기술적 토대다.

## 성숙도 모델: 우리 팀은 어디쯤인가

지금 SDD 논의에서 가장 유용한 프레임은 **성숙도 3단계**다. 팀이 스펙과 코드의 관계를 어디까지 강제하느냐로 나뉜다.

```
Spec-First  ────────►  Spec-Anchored  ────────►  Spec-as-Source
(느슨함)                (대부분 여기)                (아직 이상향)
```

| 단계 | 스펙-코드 관계 | 누가 코드를 고치나 | 적합한 곳 |
|---|---|---|---|
| **Spec-First** | 스펙으로 초기 생성, 이후 코드가 갈라져도 됨 | 사람이 코드 직접 수정 | 프로토타입, 1회성 기능 |
| **Spec-Anchored** | 스펙과 코드가 함께 진화, 테스트로 정합성 강제 | 스펙·코드 양쪽 | **대부분의 프로덕션 팀** |
| **Spec-as-Source** | 사람은 스펙만 수정, 코드는 완전 생성 | 사람은 스펙만, 코드는 손대지 않음 | 아직 실험 단계 |

핵심은 **대부분의 팀에게 정답은 spec-anchored**라는 것이다. 스펙을 유일한 진실로 삼되(spec-as-source), 코드를 절대 손대지 못하게 하는 건 아직 무리다. 스펙과 코드가 나란히 가고, 그 정합성을 **자동화된 테스트가 감시**하는 게 현실적인 균형점이다.

Tessl이 밀고 있는 spec-as-source(코드는 재생성물, 스펙만 관리)는 매력적이지만 — "같은 스펙으로 다른 기술 스택으로 갈아탄다"는 꿈까지 있다 — 아직 대부분의 조직엔 이르다.

## 실제로 "스펙을 받아서" 개발되는 사례

사용자가 궁금해한 부분이다. 정리하면 "스펙을 받는다"에는 두 가지 결이 있다.

- **(A) 조직 내부**: 기획·도메인 전문가가 스펙을 쓰고 → AI 에이전트가 구현한다. 스펙이 사람과 AI 사이의 **인터페이스**가 된다.
- **(B) 스펙이 산출물/계약**: 스펙 자체가 리뷰 게이트를 통과해야 하는 공식 산출물이 되고, 구현은 그로부터 파생된다.

가장 구체적인 실제 사례는 **AWS가 Kiro로 만든 신약 개발(타깃 발굴) 에이전트**다. 숫자가 분명하다.

```
프로젝트: 신약 타깃 발굴 에이전트 (프로덕션급)
팀:       솔루션 아키텍트 3명
기간:     3주
통합:     30개+ 외부 API

Day 1-3   요구사항 + 이해관계자 정렬  → requirements.md
Day 4-6   기술 설계/아키텍처          → design.md
Day 7-14  스펙 작성 + 풀스택 구현     → tasks.md → 구현
Day 14-21 테스트, 피드백, 반복

결과:
  · Kiro가 비즈니스 로직 코드의 95%+ 생성
  · 80시간+ 절감 (README 문서화만 8시간 추가 절감)
```

이 사례가 "스펙을 받아 개발"의 전형인 이유는 흐름에 있다.

1. **제약·바이오 조직과 협업으로 use-case를 정의**한다. 즉, 도메인 전문가의 요구가 출발점이다.
2. 그 요구가 `requirements.md`(비즈니스 요구·수용 기준·성공 지표)로 **캡처**된다.
3. `design.md` → `tasks.md`로 내려가며, **각 단계 사이에 이해관계자 리뷰**가 들어간다.
4. 사람은 스펙을 검토·정제하고, **구현의 95%는 AI가** 한다.

핵심은 개발자가 "이걸 어떻게 짤까"에 시간을 쓰지 않았다는 점이다. 도메인 전문가의 의도가 구조화된 스펙으로 들어가고, 개발자는 **스펙 리뷰와 검증**에 집중했다. 스펙이 도메인 전문가·아키텍트·AI가 만나는 수렴점이 된 것이다.

여기에 Kiro의 **steering 문서**(product.md / structure.md / tech.md)가 프로젝트 전반의 맥락을 상시 제공해서, 매번 컨벤션을 다시 설명하지 않아도 일관성이 유지됐다. 4월 글에서 말한 spec-kit의 constitution.md와 같은 발상이다.

---

## 스펙이 "계약"이 된다는 것

이 사례들이 보여주는 더 큰 그림은, 스펙이 **인터페이스이자 계약**이 되고 있다는 것이다.

기존에는 기획서와 코드 사이에 늘 간극이 있었다. 기획은 문서로 남고, 진실은 코드에 있고, 둘은 시간이 지나면 어긋났다. SDD가 바꾸는 건 이 순서다. **요구가 바뀌면 스펙을 먼저 고치고, 구현이 거기서 파생된다.** 그래서 모든 이해관계자 — 비즈니스, PM, 개발자, AI 에이전트 — 가 같은 문서를 본다.

효과는 통합 지점에서 특히 크게 나타난다. 한 사례에서는 API 변경의 사이클 타임이 **75% 줄었다**고 보고됐다. 비호환이 프로덕션이 아니라 **스펙 리뷰 단계에서** 잡혔기 때문이다.

바뀌는 건 사람의 노동 형태다.

```
[기존]  요구 이해 → 설계 → 타이핑(대부분의 시간) → 디버깅
[SDD]   요구 이해 → 스펙 명료화(대부분의 시간) → AI 구현 → 스펙 리뷰
```

노력이 "코드를 치는 일"에서 "스펙을 명료하게 만들고 리뷰하는 일"로 이동한다.

## 냉정하게: 아직 과장된 부분

확산 중이라고 해서 무결한 방법론은 아니다. 회의론도 분명히 존재한다.

- **ThoughtWorks 기술 레이더는 SDD를 "Assess" 링에 뒀다.** "Adopt"가 아니다. 흥미롭지만 워크플로우가 지나치게 정교하고 opinionated하다는 평가다.
- **Martin Fowler 진영(Birgitta Böckeler)의 지적**: 정교한 절차가 오히려 "통제하고 있다는 착각(false sense of control)"을 준다. 앞단에 스펙을 잔뜩 설계하는 것이 반복적(iterative) 개발 원칙과 충돌하고, 과거 실패한 Model-Driven Development의 데자뷰라는 우려도 있다.
- **용어의 의미 희석(semantic diffusion)**: "SDD"가 그냥 "자세한 프롬프트"의 동의어로 쓰이는 경우가 늘고 있다. 마크다운 파일 몇 개 만든 걸 SDD라고 부르는 것과, 스펙을 진짜 진실의 원천으로 강제하는 것은 다르다.

벤더가 내놓는 "3~10배 성공률" 같은 숫자도 걸러 들어야 한다. 좀 더 일관되게 관찰되는 사실은 화려한 배수가 아니라, **사람의 일이 타이핑에서 스펙 리뷰로 옮겨간다**는 질적 변화 쪽이다.

---

## 그래서 지금 어떻게 접근할까

1. **성숙도 모델에서 우리 팀 위치부터 정한다.** 대부분은 spec-anchored를 목표로 하면 된다. spec-as-source에 지금 다 걸지는 말 것.
2. **도구는 이미 쓰는 에이전트에 맞춘다.** Claude Code면 spec-kit이나 내장 SDD 커맨드, VS Code/AWS 생태계면 Kiro. 새 도구를 위한 새 도구를 도입하지 말 것.
3. **"스펙을 받는" 입장이라면 구조화된 표기로 모호성을 제거한다.** EARS 같은 트리거-응답 문장, 그리고 요구→설계→작업 각 단계 사이의 리뷰 게이트. AWS 사례가 성공한 건 도구가 아니라 이 **단계별 리뷰 규율** 덕이 크다.
4. **문화가 도구보다 먼저다.** 진짜 생산성을 얻는 조직은 최고의 도구를 가진 곳이 아니라, AI 에이전트를 "명령 대상"이 아니라 "협업자"로 대하고 스펙을 함께 만드는 문화 전환을 이룬 곳이다.

---

## 정리

4월에 "먼저 문서, 그다음 코드"라고 썼다. 3개월 뒤의 한 줄 요약은 이렇다.

> **스펙은 이제 문서가 아니라, 사람과 AI가 함께 서명하는 계약이다.**

도메인 전문가의 요구가 구조화된 스펙으로 캡처되고, AI가 그로부터 프로덕션의 95%를 만들고, 사람은 스펙을 리뷰한다 — AWS의 3주짜리 신약 개발 에이전트가 이미 그 그림을 보여줬다.

물론 아직 "Assess" 링이고, 용어는 희석되고 있고, 앞단 과설계의 위험도 남아 있다. 하지만 방향은 분명하다. AI 코딩이 흔해질수록, 병목은 **"AI가 코드를 짤 수 있는가"가 아니라 "우리가 무엇을 원하는지 스펙으로 명확히 쓸 수 있는가"** 로 옮겨간다. SDD의 확산은 결국 그 병목에 대한 업계의 대답이다.
