> "그냥 nginx 하나 세워서 외부로 열면 안 돼요?"

현업에서 가장 자주 듣는 요청이다. 결론부터 말하면 **안 된다**. 단순 리버스 프록시 노출은 「개인정보의 안전성 확보조치 기준」 제6조 제2항과 ISMS-P 2.6.6 원격접근 통제에 정면으로 미달하고, 2024~2025년 Ivanti Connect Secure 제로데이(CVE-2025-0282, CVSS 9.0) 같은 사례에서 보듯 외부 노출 게이트웨이는 1순위 공격 표면이다. "공격 표면 1개 = 사고 1개"가 된다.

권고 아키텍처는 **ZTNA + Reverse Proxy(DMZ, WAF) + MDM/MAM + MFA(FIDO2 우선)** 4단 통제다. 아래에서 왜 이렇게 가야 하는지, 그리고 단계적으로 어떻게 도입하는지 정리한다.

---

## 1. Reverse Proxy 단독이 왜 부족한가

리버스 프록시(Nginx/Apache/HAProxy)는 TLS 종단·라우팅·기본 ACL을 제공하는 **한 개의 보안 계층**일 뿐이다. 자체적으로 다음을 제공하지 않는다.

- 인증/MFA/SSO
- 단말 신뢰도(Device Posture) 평가
- 사용자별 정책·UEBA
- DLP·워터마크·캡처 차단

여기에 어플라이언스 자체 제로데이가 터지면(Ivanti·Fortinet·Pulse Secure 반복 사례) 즉시 침해로 직행한다. ID/PW만으로 인터넷에 웹시스템을 노출하면 ISMS-P 인증·개인정보위 점검에서 결함으로 잡힌다.

**ngrok/frp/Cloudflare Tunnel은?**
- ngrok·frp: 개발·데모용. 운영·로깅·접근통제 거버넌스 부족. 사내 표준 도입 대상 아님.
- Cloudflare Tunnel **+ Access** 조합은 다르다. 이건 엔터프라이즈 ZTNA로 검토 가능.

## 2. 법규가 요구하는 것

```
「개인정보의 안전성 확보조치 기준」 (개인정보위 고시 2023-6호)
─ 제6조 제2항
  외부에서 개인정보처리시스템에 접속 시
  → 인증서, 보안토큰, 일회용 비밀번호 등 "안전한 인증수단"
  → 또는 VPN 등 "안전한 접속수단"

ISMS-P 2.6.6 원격접근 통제
─ 원칙적 금지, 불가피한 경우
  → 책임자 승인
  → 접근 단말 지정
  → 강화된 인증 + 구간 암호화
  → 접속단말 보안(백신·패치)
```

위반 시 「개인정보 보호법」 제29조 안전조치 의무 위반 → 5천만원 이하 과태료, 사고 발생 시 전체 매출액 3% 이하 과징금.

산업별 추가 규제도 본다. 금융은 「전자금융감독규정」 망분리 조항(현재 단계적 완화 중, 2024.8.13 로드맵 발표·2026.1.20 시행세칙 개정안 사전예고), 의료는 「의료법」, 공공은 「국가정보보안기본지침」.

## 3. 권고 아키텍처: 4단 통제

```
[모바일 단말]
     │  (1) MDM/MAM 등록 + 단말 인증서
     │      Knox/Intune/Jamf, FLAG_SECURE, 워터마크
     ▼
[ZTNA 클라이언트 또는 SSL VPN+조건부 접근]
     │  (2) ID/디바이스/컨텍스트별 매 요청 검증
     │      애플리케이션 단위 접근, 인터넷 미노출
     ▼
[DMZ: Reverse Proxy + WAF]
     │  (3) TLS 1.3 종단, OWASP CRS, Rate Limit,
     │      API 스키마 검증, Bot 관리
     ▼
[IdP: SAML/OIDC + MFA]
     │  (4) FIDO2/패스키 우선, OTP 차선
     │      SMS OTP는 SIM swap 위험 — 비권고
     ▼
[내부 웹시스템]
```

NIST SP 800-207(Zero Trust)·800-46 Rev.2(원격접근)·800-124 Rev.2(모바일) 권고와 일치하고, ISMS-P 2.5.3/2.6.6/2.6.7/2.10.6 + 개인정보 안전성 확보조치 제5~8조를 통합 충족한다.

### 솔루션 옵션 비교

| 방식 | 보안성 | 모바일 UX | 비용 | 운영 부담 | 시나리오 |
|---|---|---|---|---|---|
| Reverse Proxy 단독 | ★ | ★★★★ | 저 | 중 | **비추천** |
| Reverse Proxy + WAF + MFA | ★★★ | ★★★★ | 중 | 중 | 공개 정보 위주 |
| SSL VPN | ★★ | ★★ | 중 | 중 | 레거시 호환 |
| **ZTNA(SaaS)** | ★★★★★ | ★★★★ | 중~고 | 저 | **대부분 케이스 권장** |
| ZTNA(자체호스팅) | ★★★★ | ★★★★ | 고 | 고 | 클라우드 불가 환경 |
| VDI/DaaS | ★★★★★ | ★★ | 고 | 고 | 고민감 + BYOD |
| 망연계 + 전용앱 | ★★★★ | ★★★ | 매우 고 | 고 | 공공/금융/국방 |

ZTNA 주요 제품: Cloudflare Access(One), Zscaler ZPA, Cisco Secure Access, Palo Alto Prisma Access, Netskope Private Access, Twingate, Tailscale. 자체호스팅은 Pomerium·OpenZiti.

## 4. 모바일 단말 통제 — PC와 다른 위협

OWASP Mobile Top 10 2024(8년 만의 개정)는 다음을 강조한다.

- **M1** Improper Credential Usage
- **M2** Inadequate Supply Chain Security
- **M3** Insecure Authentication/Authorization
- **M9** Insecure Data Storage

대응 통제는 다음 조합이 표준이다.

| 영역 | 통제 |
|---|---|
| 화면 캡처 | Android `FLAG_SECURE`, iOS `isCaptured` + 워터마크 |
| 클립보드/공유 | MAM/Work Profile로 컨테이너 격리 |
| 단말 무결성 | Knox Attestation, Google Play Integrity, Apple App Attest |
| 루팅/탈옥 | MDM compliance + MTD(Mobile Threat Defense) |
| 분실/도난 | 원격 잠금·원격 와이프(킬 스위치) |

**BYOD vs 회사지급**
- **COBO**(회사지급): 통제 최대, TCO 최고
- **COPE**(회사소유+개인사용): Work Profile, 균형형
- **BYOD**: MAM-only 권장 — MDM이 개인 영역에 접근하면 「개인정보보호법」 동의 이슈 발생

**모바일 앱 vs 모바일 웹**: 고민감/정기 사용은 네이티브 앱(SSL Pinning·RASP·FLAG_SECURE), 저~중위험/간헐 사용은 반응형 웹.

## 5. 시스템 민감도별 차등 적용

모든 사내 시스템에 일괄 허용하면 안 된다. 다음 기준으로 분류한다.

| 위험 등급 | 시스템 예시 | 권고 |
|---|---|---|
| 매우 높음 | 주민번호·민감정보 대량 처리, 결제 승인, 핵심 R&D | **외부 접속 금지** 또는 VDI 화면전송만 |
| 높음 | 인사·급여·계약, 관리자 콘솔 | ZTNA + FIDO2 + COBO + MDM, 로그 2년 |
| 중간 | 결재·그룹웨어·문서뷰어 | ZTNA/SSL VPN + MFA + MDM/MAM + 워터마크 |
| 낮음 | 공지·사내 위키(비기밀)·근태 | RP + WAF + MFA, BYOD-Work Profile 허용 |

**절대 외부로 열지 말아야 할 시스템**
- 일평균 100만명 이상 처리자의 인터넷망 차단 의무 단말(고시 제6조 제6항)
- 결제·인증·KMS/HSM 관리 콘솔
- 기간계(코어뱅킹·기간계 ERP) 운영 콘솔
- 소스코드 저장소·CI/CD 핵심·설계도 DB (공급망 공격)
- 임직원 주민번호·민감정보 일괄 조회 화면

## 6. 도입 로드맵 (12~18개월)

```
Phase 0  사전 준비          1~2개월
         자산 목록·위험평가·민감도 분류
         정보보호위원회 안건 상정

Phase 1  PoC                2~3개월
         저~중위험 시스템 1~2개 (그룹웨어/결재)
         50~100명 한정
         ZTNA + MFA + MDM(BYOD-Work Profile)
         KPI: 인증 성공률, 응답시간, 만족도

Phase 2  파일럿             3~6개월
         부서 1~2개 전체 (500명)
         사고 시나리오 모의훈련 2회
         사용자 교육 완료

Phase 3  전사 확대          6~12개월
         시스템 분류별 차등 적용
         분기 권한 재검토 정착
         SIEM 룰셋 고도화

Phase 4  성숙도 강화
         ZTNA Continuous Verification
         FIDO2 전면, MTD 도입
```

**롤백 기준**
- 인증 실패율 5% 초과 지속 → MFA UX 재설계
- 보안 사고 1건 이상 → 영향 시스템 격리·복귀 보류
- 운영 비용 예상치 130% 초과 → 솔루션 재선정

## 7. 운영·거버넌스 체크리스트

| 항목 | 요구 사항 |
|---|---|
| 인증 | ID/PW + MFA 의무 (FIDO2 우선) |
| 암호화 | TLS 1.2 이상, HSTS, 약한 cipher 금지 |
| 단말 | MDM/MAM 등록 필수, 미등록 차단 |
| 로그 | 1년(5만명·민감정보·기간통신은 2년), 월 1회 점검 |
| 권한 | 분기 access review, 90일 미사용 비활성화 |
| 거버넌스 | 정보보호위원회 의결, CISO/CEO 위험 수용 서명 |
| 패치 SLA | Critical CVE 24~72시간 |
| 사고 대응 | KISA 118 신고, 개인정보 유출 시 72시간 내 정보주체 통지 |

## 마무리

핵심은 두 가지다.

1. **"리버스 프록시로 그냥 열면 된다"는 인식부터 깨야 한다.** 리버스 프록시는 한 계층일 뿐이고, 인증·단말·DLP·감사를 통합한 4단 통제가 최소 기준선이다.
2. **시스템 민감도별로 다르게 간다.** 모든 시스템을 똑같이 처리하면 보안과 사용성 둘 다 잃는다. 매우 높음은 막고, 낮음은 열고, 그 사이는 ZTNA로 푼다.

"망분리 완화 = 통제 완화"가 아니라 **"통제의 재구성"** 이다. 4단 통제와 거버넌스 문서화 없이 외부 접속을 허용하면, 사고 시 책임 소재가 곧장 CISO·CEO로 향한다는 점을 잊지 말자.
