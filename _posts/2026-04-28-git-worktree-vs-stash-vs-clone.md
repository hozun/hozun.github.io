> "5분만 다른 브랜치에서 확인해야 하는데..." 이 한 마디로 시작되는 컨텍스트 스위칭. 어떻게 처리하는 게 가장 효율적일까.

---

## 결론부터

| 상황 | 추천 |
|---|---|
| 5분 안에 끝낼 짧은 확인, 빌드 영향 없음 | **stash** |
| 두 브랜치를 며칠~몇 주 동안 병행 작업 | **worktree** |
| 다른 remote/포크/완전히 격리된 환경 필요 | **별도 clone** |

핵심 기준은 **얼마나 자주 오갈 것인가**, **빌드 산출물을 공유할 수 있는가**, **격리 수준이 얼마나 필요한가** 세 가지다.

---

## 세 가지 방식 한눈에 보기

```
[같은 저장소, 같은 워킹 디렉토리]
  ┌──────────────────────────┐
  │  branch A  ⇄  branch B   │   ← stash
  │  (한 번에 하나만 체크아웃)│
  └──────────────────────────┘

[같은 저장소, 다른 워킹 디렉토리들]
  ┌──────────────────┐  ┌──────────────────┐
  │ /repo  branch A  │  │ /repo-b branch B │   ← worktree
  └──────────────────┘  └──────────────────┘
            └──── .git 공유 ────┘

[독립된 저장소]
  ┌──────────────────┐  ┌──────────────────┐
  │ /repo  (독립 .git)│  │ /repo2 (독립 .git)│   ← 별도 clone
  └──────────────────┘  └──────────────────┘
       fetch 따로            fetch 따로
```

---

## 1. git stash

**무엇:** 워킹 디렉토리의 변경사항을 임시로 스택에 저장하고 깨끗한 상태로 만든다.

```bash
git stash push -m "WIP: 결제 모듈 리팩터링"
git checkout hotfix/login-bug
# 작업 후
git checkout feature/refactor-payment
git stash pop
```

**장점**
- 가볍고 빠르다. 디스크 추가 사용 없음.
- 같은 워킹 디렉토리에서 그대로 작업 → IDE 인덱스, 터미널, 디버거 그대로.

**단점**
- 브랜치 전환 시 **빌드 산출물이 무효화**된다. `node_modules` 호환 안 되는 의존성 차이, `.next` / `target/` / `build/` 다 새로 빌드.
- stash가 쌓이면 뭐가 뭔지 잊어버린다.
- 충돌 가능성 — pop 시 conflict 나면 풀기 귀찮다.
- 추적되지 않는 새 파일은 `-u` 안 붙이면 stash에 안 들어간다.

**적합한 상황**
- "지금 1분만 main에서 로그 한 줄 확인" 수준의 짧은 끼어들기
- 빌드 시간이 짧은 프로젝트
- 변경량이 작아 conflict 위험이 낮을 때

---

## 2. git worktree

**무엇:** 한 저장소를 여러 디렉토리에 동시에 체크아웃한다. `.git`은 공유, 워킹 디렉토리만 분리.

```bash
# 메인 저장소: ~/work/repo (feature 브랜치)
cd ~/work/repo
git worktree add ../repo-hotfix hotfix/login-bug
# 새 워크트리 생성 — ~/work/repo-hotfix 에서 hotfix 브랜치 체크아웃됨

cd ../repo-hotfix
# 여기서 hotfix 작업, 빌드, 테스트 — 메인 디렉토리 영향 X

# 작업 끝나면 정리
cd ../repo
git worktree remove ../repo-hotfix
```

**장점**
- 두 브랜치를 **동시에** 작업할 수 있다. IDE/에디터 두 개 띄워놓고 왔다갔다.
- 빌드 산출물(`node_modules`, `.next`, `target/`)이 디렉토리별로 분리 → 재빌드 안 함.
- `.git`은 하나라서 fetch 한 번이면 모든 워크트리에 반영. 디스크 효율 좋음.
- stash보다 안전 — 변경사항이 워킹 디렉토리에 그대로 남아있다.

**단점**
- 같은 브랜치를 두 워크트리에서 동시 체크아웃 불가 (기본 설정).
- 워크트리 관리 필요. `git worktree list`, `git worktree prune` 같은 명령을 알아야 함.
- 디렉토리가 늘어나니 IDE 워크스페이스 설정도 늘어남.

**적합한 상황**
- PR 리뷰하면서 본인 작업도 계속 진행
- hotfix 들어왔는데 진행 중인 feature를 stash로 묻기엔 너무 큼
- 장기 브랜치(릴리스 브랜치 등)와 메인 작업을 병행
- 빌드가 오래 걸리는 프로젝트 (Rust, 큰 React 프로젝트, monorepo)

---

## 3. 별도 디렉토리에 clone

**무엇:** 같은 원격에서 다시 `git clone`해서 완전히 독립된 저장소를 만든다.

```bash
cd ~/work
git clone git@github.com:org/repo.git repo-experiment
```

**장점**
- 완전한 격리. 다른 저장소가 어떤 영향도 받지 않는다.
- 다른 remote, 다른 user.email, 다른 hook 설정 가능.
- 한쪽에서 `git reset --hard` / `rm -rf .git` 해도 다른 쪽은 무사.

**단점**
- **디스크 사용 2배**. 큰 모노레포에서는 큰 부담.
- `fetch` / `pull`을 따로 해야 한다. 한쪽에서 받은 커밋이 다른 쪽에 자동 반영 안 됨.
- 같은 작업 환경 세팅을 두 번 해야 함 (env 파일, 의존성 설치 등).

**적합한 상황**
- 다른 포크나 다른 remote를 쓸 때 (예: upstream 따라가는 fork 작업)
- 완전히 다른 브랜드/제품 라인 (예: 같은 코드베이스인데 클라이언트별 커스터마이즈)
- 실험적인 큰 변경을 한 디렉토리에서 하고 싶고, 메인은 절대 영향 안 받게 하고 싶을 때
- 다른 SSH 키나 Git 사용자 정보를 써야 할 때 (회사 vs 개인)

## 비교표

| 항목 | stash | worktree | 별도 clone |
|---|---|---|---|
| 디스크 사용 | 거의 없음 | 워킹 디렉토리만 추가 | 저장소 전체 × N |
| `.git` 공유 | (단일 저장소) | 공유 | 독립 |
| 빌드 산출물 재사용 | X (브랜치 전환 시 무효) | O (디렉토리별 유지) | 각자 빌드 |
| 동시 작업 | 불가 | 가능 | 가능 |
| fetch 한 번에 반영 | (단일) | O | X (각자) |
| 격리 수준 | 낮음 | 중간 | 높음 |
| 셋업 비용 | 0 | 명령 한 줄 | clone + 환경 셋업 |
| 실수 시 영향 | 같은 디렉토리 위험 | 디렉토리 분리됨 | 완전 격리 |

## 실전 의사결정 흐름

```
다른 브랜치 작업이 필요한가?
        │
        ├─ 5분 안에 끝나고 빌드 영향 없음
        │       → stash
        │
        ├─ 며칠 동안 두 브랜치 병행
        │       → worktree
        │
        ├─ 다른 remote / 포크 / 완전 격리
        │       → 별도 clone
        │
        └─ 애매하면 worktree
              (디스크 여유만 있으면 가장 균형 좋음)
```

---

## worktree 자주 쓰는 명령어 모음

```bash
# 현재 워크트리 목록
git worktree list

# 새 브랜치를 새 워크트리로 바로 생성
git worktree add -b feature/new-thing ../repo-new main

# 기존 브랜치를 워크트리로
git worktree add ../repo-hotfix hotfix/login-bug

# 워크트리 삭제 (디렉토리 정리됨)
git worktree remove ../repo-hotfix

# 디렉토리를 수동으로 지웠을 때 등록 정리
git worktree prune
```

---

## 정리

- **stash**: 짧은 끼어들기, 가볍지만 위험 있음.
- **worktree**: 대부분의 컨텍스트 스위칭에 가장 균형 좋은 선택. 한 번 익히면 stash를 거의 안 쓰게 됨.
- **별도 clone**: 진짜 격리가 필요할 때만. 디스크/관리 비용을 감수하는 만큼 확실히 안전.

stash는 익숙해서 자주 쓰지만, worktree로 갈아타면 빌드 캐시 날아가는 스트레스부터 줄어든다. 모노레포일수록 차이가 더 크다.
