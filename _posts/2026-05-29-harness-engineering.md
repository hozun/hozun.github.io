같은 Claude Sonnet 4.6을 쓰는 도구가 시장에 수십 개 있다. Cursor, Claude Code, Aider, Cline, Continue, Windsurf… 그런데 똑같은 작업을 시켜보면 결과가 천차만별이다. 누구는 한 번에 끝내고, 누구는 엉뚱한 파일을 건드리고, 누구는 무한 루프에 빠진다.

**모델은 똑같은데 왜 다르냐.** 차이는 모델 바깥, 모델을 감싸고 있는 **하네스(harness)**에서 나온다. 그리고 이 하네스를 설계하는 일이 새로운 직무로 자리 잡고 있다 — 하네스 엔지니어링(harness engineering).

---

## 하네스가 뭔가

원래 하네스는 말이나 사람에게 채우는 마구·안전벨트를 뜻한다. AI 맥락에서는 **LLM을 둘러싸고 LLM이 실제 일을 할 수 있게 만드는 모든 인프라**를 의미한다.

쉽게 말해 이런 것들이다.

| 구성 요소 | 역할 |
|---|---|
| 시스템 프롬프트 | 모델의 성격·규칙·기본 행동 정의 |
| 툴 정의(tool schema) | 모델이 외부 세계와 상호작용하는 채널 |
| 컨텍스트 관리 | 무엇을 기억하고, 무엇을 압축하고, 무엇을 버릴지 |
| 루프 제어 | 언제 멈출지, 언제 다시 호출할지 |
| 권한·안전장치 | 어떤 행동을 자동 승인하고 어떤 건 사람한테 물을지 |
| 출력 파싱 | 모델 응답에서 무엇을 추출해 실행할지 |
| 메모리·상태 | 세션 간 무엇을 가져가고 무엇을 버릴지 |

모델은 입력이 들어오면 토큰을 뱉어내는 **함수**일 뿐이다. 그 함수 호출을 어떤 순서로, 어떤 컨텍스트와 함께, 어떤 도구를 쥐어주고 부르느냐 — 이걸 설계하는 게 하네스다.

---

## 왜 모델보다 하네스가 중요해졌나

2024년까지만 해도 업계 분위기는 "모델만 좋아지면 다 해결된다"였다. 실제로 GPT-4가 나왔을 때 많은 RAG 라이브러리들이 그냥 쓸모없어졌다. 모델이 충분히 똑똑해지니까 잔재주가 필요 없었다.

그런데 2025~2026년 들어서 양상이 바뀌었다.

**첫째, 최상위 모델들의 성능이 평준화됐다.** Claude 4.7, GPT-5.x, Gemini 3.x — 일반 코딩 벤치마크에서는 다 비슷비슷하다. 모델 선택만으로 차이를 만들기 어렵다.

**둘째, 에이전트 워크플로의 복잡도가 폭발했다.** 단발 질의응답이면 모델만 좋으면 된다. 그런데 "코드베이스 분석 → 계획 수립 → 파일 수정 → 테스트 실행 → 실패 시 디버깅 → 커밋"처럼 여러 턴이 얽히면, **모델 호출 사이사이의 설계**가 성능을 좌우한다.

**셋째, 같은 모델을 쓰는 도구들 사이에 실제 성능 격차가 보였다.** SWE-bench 같은 벤치마크에서 동일 모델로 다른 하네스를 돌리면 점수가 10~20%P씩 갈린다. 이 시점부터 "벤치마크는 모델이 아니라 하네스+모델 조합을 평가한다"는 게 상식이 됐다.

---

## 같은 모델, 다른 하네스 — 실제 예시

Claude Code, Cursor, Aider는 모두 Claude를 쓸 수 있다. 모델은 같아도 하네스가 달라 결과가 달라진다.

```
[Claude Code]
- 풀 디렉토리 컨텍스트, Bash 직접 실행
- 시스템 프롬프트에 "병렬 도구 호출 우선" 규칙
- 자동 컨텍스트 압축 + 메모리 시스템
→ 다중 파일 리팩토링, 장시간 작업에 강함

[Cursor]
- 에디터 통합, 변경 즉시 diff로 시각화
- 파일/심볼 단위 좁은 컨텍스트 + 시맨틱 인덱싱
- 사용자 승인 루프가 더 빈번
→ 한 파일 안 정밀한 편집, 인터랙티브 작업에 강함

[Aider]
- 깃 커밋 단위로 변경 추적
- 영향 파일을 사용자가 명시적으로 add
- repo-map으로 압축된 코드베이스 요약 주입
→ 의도가 분명한 작은 변경, 깃 기반 워크플로에 강함
```

세 도구 모두 좋은 도구지만 강점이 다르다. 그 차이의 90%는 모델이 아니라 하네스 설계에서 온다.

---

## 하네스 엔지니어가 실제로 다루는 문제들

추상적인 얘기 말고 구체적으로 어떤 결정들을 해야 하는지 보면 감이 온다.

**컨텍스트 윈도우 운영**. 200K 토큰 컨텍스트가 있다고 다 채우면 안 된다. 모델은 컨텍스트가 길어질수록 중간 부분을 놓치는 lost-in-the-middle 현상이 있고, 비용·지연도 같이 늘어난다. 어디까지가 "현재 작업에 필요한 최소한"인가를 정해야 한다.

**툴 디자인**. `read_file(path)` 하나를 줄지, `read_file(path, line_range)`을 줄지, `grep(pattern, path)`까지 줄지에 따라 모델 행동이 바뀐다. 툴이 많으면 모델이 헷갈리고, 적으면 비효율적인 우회 행동을 한다. 적절한 추상화 수준을 찾는 게 일이다.

**시스템 프롬프트 vs 도구 응답 vs 사용자 메시지**. 같은 정보라도 어디에 넣느냐로 모델 반응이 달라진다. 지속적으로 따라야 할 규칙은 시스템 프롬프트, 도구 결과에 자연스럽게 따라오는 지침은 도구 응답, 일회성 요청은 사용자 메시지에 넣는다.

**루프 종료 조건**. 에이전트는 "다 됐다"고 판단하기를 자주 틀린다. 너무 일찍 멈추거나, 끝없이 자기 검증을 반복한다. 명시적인 종료 신호, 최대 턴 수 제한, 외부 검증(테스트 통과, 빌드 성공) 등을 어떻게 묶을지 설계가 필요하다.

**사람 개입 지점**. 무엇을 자동 승인하고 무엇을 물어볼지. 너무 자주 물어보면 사람이 지치고, 안 물어보면 사고가 난다. "되돌리기 어려운 작업"과 "로컬에서 반복 가능한 작업"을 구분하는 정책 자체가 하네스의 일부다.

## 프롬프트 엔지니어링과의 차이

프롬프트 엔지니어링과 하네스 엔지니어링은 헷갈리기 쉬운데, 차이는 명확하다.

| | 프롬프트 엔지니어링 | 하네스 엔지니어링 |
|---|---|---|
| 단위 | 단발 호출 | 다중 호출·루프 |
| 산출물 | 텍스트(프롬프트) | 시스템(코드+프롬프트+툴+정책) |
| 평가 | 응답 품질 | 작업 성공률·비용·지연·안전성 |
| 주 도구 | LLM 플레이그라운드 | SDK, 평가 프레임워크, 트레이스 분석 |
| 비유 | 한 마디 잘 던지기 | 직원 한 명 셋업하기 |

프롬프트 엔지니어링은 하네스 엔지니어링의 부분집합이다. 좋은 프롬프트는 여전히 중요하지만, 그것만으로는 안 된다는 게 합의된 관점이다.

---

## 평가가 어렵다

하네스가 성능을 결정한다는 건 평가가 어렵다는 뜻이기도 하다. 모델 벤치마크 점수는 더 이상 충분한 정보가 아니다. "Claude X.X가 SWE-bench에서 N%"라는 숫자는 항상 **어떤 하네스로 측정했느냐**가 따라붙어야 의미가 있다.

실무에서는 모델 단위가 아니라 **하네스+모델 조합 단위로 자체 벤치마크를 돌리는 팀이 늘고 있다**. 자기 도메인의 실제 태스크를 50~200개 모아두고, 하네스를 바꿀 때마다 전체를 돌려본다. 트레이스(모델 입출력 전체 로그)를 저장해두고 회귀를 분석한다. 평가 인프라 자체가 하네스 엔지니어의 핵심 결과물이 된다.

---

## 마무리

모델 성능이 평준화될수록 하네스가 차별화 포인트가 된다. AI 도구를 만드는 쪽이라면 모델 선정만큼이나 하네스 설계에 시간을 써야 하고, AI 도구를 쓰는 쪽이라면 같은 모델이라도 하네스가 다른 도구는 다른 도구라는 점을 인식해야 한다.

"이 모델 좋더라" 보다 **"이 하네스에서 이 모델 조합이 좋더라"**가 더 정확한 표현이 된 시대다.
