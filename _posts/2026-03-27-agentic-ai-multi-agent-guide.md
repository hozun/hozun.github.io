요즘 기술 뉴스를 보면 'AI 에이전트', '멀티에이전트', 'Agentic AI' 같은 단어가 넘쳐난다. 그런데 막상 "그게 뭔데요?"라고 물으면 명확하게 대답하기 쉽지 않다. 개념이 겹치는 것 같고, 정확히 어디서 어떻게 쓰이는지 잘 안 잡힌다.

이 글에서 한 번에 정리한다. 그냥 AI와 Agentic AI의 차이부터 시작해서, Multi-Agent 패턴, 핵심 용어 사전까지. 초중급 개발자 눈높이에서 최대한 구체적으로.

## 1. 그냥 AI와 Agentic AI, 뭐가 다른가

기존 AI의 동작 방식은 단순하다. 내가 질문하면 AI가 답한다. 끝. 다음 질문이 올 때까지 AI는 아무것도 안 한다. 완전히 **반응형(Reactive)**이다.

ChatGPT에 "제주도 2박 3일 여행 계획 짜줘"라고 하면, 그럴듯한 일정표를 만들어준다. 하지만 실제로 항공권을 검색하지 않는다. 호텔을 비교하지 않는다. 예약은 당연히 안 된다. 텍스트로 된 계획서만 나올 뿐이다.

**Agentic AI**는 다르다. 목표를 주면 스스로 계획을 세우고, 필요한 도구를 호출하고, 결과를 확인하고, 다음 단계를 결정한다. 멈추라고 하거나 목표를 달성할 때까지 루프를 돌면서 일을 처리한다. **자율형(Autonomous)**이다.

핵심 차이는 세 가지다.

**도구 사용(Tool Use)** — Agentic AI는 외부 시스템을 직접 호출할 수 있다. 검색 엔진을 쿼리하고, API를 호출하고, 파일을 읽고 쓴다. 기존 AI는 학습된 지식으로만 답한다.

**기억(Memory)** — 대화 맥락(단기 기억)은 물론, 벡터 DB 같은 외부 저장소에 정보를 저장하고 나중에 꺼내 쓸 수 있다(장기 기억). 과거 작업의 결과를 활용해 더 나은 판단을 내린다.

**루프 실행(ReAct loop)** — 한 번 답하고 끝나는 게 아니라, 목표가 달성될 때까지 생각 → 행동 → 관찰을 반복한다. 중간에 실수를 발견하면 스스로 수정한다.

## 2. AI 에이전트의 4가지 핵심 요소

에이전트가 어떻게 돌아가는지 구조적으로 이해하려면 네 가지 요소를 알아야 한다.

### Perception (인식)

에이전트가 환경을 읽는 단계다. 파일 시스템, 웹페이지, 데이터베이스, API 응답... 에이전트가 "지금 상황이 어떻다"는 걸 파악하기 위해 정보를 수집한다. MCP 서버 같은 도구가 이 역할을 담당하는 경우가 많다.

### Planning (계획)

목표를 받으면 에이전트가 "어떻게 달성할까"를 스스로 설계한다. 큰 목표를 작은 단계로 쪼개고, 어떤 도구를 어떤 순서로 쓸지 결정한다. LLM의 추론 능력이 여기서 발휘된다.

### Action (실행)

계획에 따라 실제로 도구를 호출한다. 웹 검색, 코드 실행, 파일 작성, API 호출... 에이전트가 "말"이 아니라 "행동"을 하는 단계다.

### Memory (기억)

**단기 기억(Short-term)** 은 현재 대화나 작업의 context다. LLM의 context window 안에 들어있는 정보가 여기 해당한다.

**장기 기억(Long-term)** 은 context window 밖에 저장해두는 정보다. 벡터 DB(Pinecone, Weaviate 등)에 임베딩으로 저장해두고, 필요할 때 유사도 검색으로 꺼내 쓴다. "지난주에 작성했던 보고서 스타일로 써줘" 같은 게 가능해진다.

## 3. ReAct 패턴 — 에이전트가 생각하는 방식

**ReAct**는 **Re**asoning + **Act**ing의 합성어다. 에이전트가 작업을 처리하는 핵심 루프 패턴이다.

흐름은 이렇다.

```
Thought: 지금 상황을 파악하고 다음에 뭘 해야 할지 생각한다
Action: 필요한 도구를 호출한다
Observation: 도구 호출 결과를 확인한다
→ 목표 달성? → 완료
→ 아직? → 다시 Thought로
```

코드로 보면 대략 이런 구조다.

```python
def react_loop(goal, tools, max_steps=10):
    context = f"목표: {goal}\n"

    for step in range(max_steps):
        # LLM이 생각하고 다음 행동 결정
        thought_and_action = llm.think(context)

        if thought_and_action.is_done():
            return thought_and_action.final_answer

        # 도구 실행
        tool_name = thought_and_action.tool
        tool_input = thought_and_action.input
        observation = tools[tool_name].run(tool_input)

        # 결과를 context에 추가
        context += f"\nThought: {thought_and_action.thought}"
        context += f"\nAction: {tool_name}({tool_input})"
        context += f"\nObservation: {observation}"

    raise Exception("최대 단계 초과")
```

왜 이게 중요한가? LLM이 **실수를 스스로 발견하고 수정**할 수 있기 때문이다.

예를 들어 에이전트가 파일을 열려고 했는데 "파일이 없음" 에러가 난다. Observation에서 이를 확인한 에이전트는 다음 Thought에서 "파일 경로가 틀렸구나, 먼저 디렉토리 목록을 확인해야겠다"고 판단하고 다른 행동을 취한다. 한 번 실패해도 포기하지 않는다.

이 루프가 없으면 AI는 한 번 틀리면 끝이다. ReAct 덕분에 에이전트가 "복잡한 작업"을 처리할 수 있게 된다.

## 4. Multi-Agent — 왜 에이전트가 여러 개 필요한가

단일 에이전트로도 많은 걸 할 수 있다. 그런데 왜 여러 에이전트를 쓰는 걸까?

이유는 세 가지다.

**Context Window 한계** — LLM은 한 번에 처리할 수 있는 텍스트 양에 한계가 있다. 복잡한 프로젝트 전체를 하나의 에이전트가 맡으면 context가 부족해진다. 작업을 나눠 각 에이전트가 자기 몫만 맡으면 이 한계를 극복할 수 있다.

**전문화** — 모든 걸 잘하는 제너럴리스트 에이전트보다, 각자 특화된 서브에이전트들이 협력하는 게 더 효율적이다. 검색에 최적화된 에이전트, 코드 작성에 최적화된 에이전트, 문서 작성에 최적화된 에이전트를 따로 두는 것이다.

**병렬 처리** — 독립적인 작업은 동시에 처리할 수 있다. 하나의 에이전트가 순서대로 처리하면 오래 걸릴 일을, 여러 에이전트가 동시에 처리하면 빠르다.

### Orchestrator와 Sub-agent

Multi-agent 시스템에는 두 가지 역할이 있다.

**Orchestrator(오케스트레이터)** 는 전체 작업을 지휘한다. 큰 목표를 받아서 어떤 서브에이전트에게 무엇을 시킬지 결정하고, 결과를 취합해서 최종 아웃풋을 만든다. 개발팀의 PM 역할이다.

**Sub-agent(서브 에이전트)** 는 오케스트레이터가 지시한 특정 작업을 수행한다. 프론트엔드 개발자, 백엔드 개발자, QA처럼 각자 맡은 영역이 있다.

비유를 그대로 쓰면 이렇다.

```
오케스트레이터 (PM): "이번 스프린트 목표는 사용자 인증 기능 구현"
  ↓
서브에이전트1 (백엔드): "JWT 인증 API 구현"
서브에이전트2 (프론트엔드): "로그인 UI 컴포넌트 작성"
서브에이전트3 (QA): "인증 플로우 테스트 케이스 작성"
```

## 5. Multi-Agent 패턴 3가지

### Sequential (순차형)

에이전트들이 컨베이어 벨트처럼 순서대로 처리한다.

```
에이전트A (데이터 수집) → 에이전트B (분석) → 에이전트C (보고서 작성)
```

앞 에이전트의 결과가 다음 에이전트의 입력이 된다. 각 단계가 이전 단계에 의존할 때 적합하다.

### Parallel (병렬형)

독립적인 작업을 여러 에이전트가 동시에 처리하고, 결과를 취합한다.

```
오케스트레이터
  ├─ 에이전트A (국내 시장 조사) ─┐
  ├─ 에이전트B (해외 시장 조사) ──┤ → 결과 취합 → 최종 보고서
  └─ 에이전트C (경쟁사 분석) ───┘
```

서로 무관한 작업을 동시에 처리하므로 시간이 크게 줄어든다.

### Hierarchical (계층형)

오케스트레이터 아래에 서브 오케스트레이터가 있고, 그 아래에 다시 전문 에이전트들이 있는 구조다.

```
메인 오케스트레이터
  ├─ 리서치 오케스트레이터
  │    ├─ 웹 검색 에이전트
  │    └─ 문서 분석 에이전트
  └─ 개발 오케스트레이터
       ├─ 코드 작성 에이전트
       └─ 테스트 에이전트
```

복잡도가 높은 대형 태스크에 적합하다. 각 계층이 자기 영역의 복잡성을 담당한다.

## 6. 실제 사례로 보기

### 리서치 에이전트

"최신 경쟁사 분석 보고서 만들어줘"라는 요청을 받으면:

1. 오케스트레이터가 작업을 분해한다
2. 검색 에이전트가 각 경쟁사 최신 뉴스·제품 업데이트를 수집한다 (병렬)
3. 요약 에이전트가 각 수집 결과를 정리한다
4. 문서 작성 에이전트가 최종 보고서를 작성한다

사람이 하루 종일 해야 할 일을 30분 만에 처리한다.

### 코딩 에이전트

GitHub 이슈가 들어오면:

1. 오류 분석 에이전트가 스택 트레이스와 관련 코드를 분석한다
2. 수정 에이전트가 코드 패치를 작성한다
3. 테스트 에이전트가 수정 코드에 대한 테스트를 실행한다
4. 문제없으면 PR 생성 에이전트가 Pull Request를 만든다

개발자는 최종 PR을 리뷰하고 머지만 하면 된다.

Claude의 여러 AI 제품들이 실제로 이 구조로 작동하고 있다. 오케스트레이터가 전체 흐름을 관리하면서 전문 서브에이전트에게 일을 위임하는 패턴이다.

## 7. 용어 사전

자주 등장하는 핵심 용어들을 한 번에 정리한다.

| 용어 | 한 줄 정의 |
|------|-----------|
| **Agent** | 목표를 받아 스스로 계획·실행하는 AI 시스템 |
| **Agentic AI** | 자율적으로 판단하고 행동하는 AI의 특성 |
| **Orchestrator** | 멀티에이전트 시스템에서 전체 흐름을 지휘하는 에이전트 |
| **Sub-agent** | 특정 작업을 전담하는 전문 에이전트 |
| **Tool Use** | AI가 외부 함수·API·서비스를 호출하는 능력 |
| **Function Calling** | LLM이 특정 함수를 호출하도록 요청하는 방식 (Tool Use의 구현 방법 중 하나) |
| **ReAct** | Reasoning + Acting. 생각→행동→관찰 루프 패턴 |
| **Chain of Thought (CoT)** | LLM이 답하기 전에 단계적으로 추론 과정을 전개하는 방식 |
| **Short-term Memory** | 현재 context window 내의 정보. 대화가 끝나면 사라진다 |
| **Long-term Memory** | 벡터 DB 등 외부 저장소에 지속적으로 보관되는 정보 |
| **Context Window** | LLM이 한 번에 처리할 수 있는 텍스트의 최대 길이 |
| **Autonomous Agent** | 인간 개입 없이 스스로 작업을 완료하는 에이전트 |
| **Supervised Agent** | 중요한 단계마다 인간의 확인·승인을 받는 에이전트 |
| **Human-in-the-loop** | 에이전트가 특정 시점에 사람의 판단을 요청하는 설계 방식 |

## 8. 개발자로서 어떻게 시작할까

이론을 알았으니 이제 실제로 뭘 해야 할지가 중요하다.

### 프레임워크 선택

에이전트를 직접 밑바닥부터 만들 필요는 없다. 잘 만들어진 프레임워크가 이미 많다.

**LangGraph** — LangChain 팀이 만든 그래프 기반 에이전트 프레임워크. 노드와 엣지로 복잡한 에이전트 흐름을 시각적으로 설계할 수 있다. 상태 관리가 탄탄하다.

**CrewAI** — 멀티에이전트 협업에 특화된 프레임워크. 역할(Role), 목표(Goal), 백스토리(Backstory)로 에이전트를 정의하는 방식이 직관적이다.

**AutoGen** — Microsoft가 만든 멀티에이전트 프레임워크. 에이전트들이 대화하듯 협업하는 방식이 특징이다.

**Claude Agent SDK** — Anthropic 공식 SDK. Claude 모델 기반 에이전트를 만들기에 가장 자연스러운 선택이다.

### 단계별 접근

처음부터 멀티에이전트를 만들려고 하면 복잡도에 압도된다. 순서가 있다.

**1단계: 도구(Tool) 하나 붙여보기** — LLM에 함수 하나 연결해서 외부 API를 호출해보는 것. 날씨 API 붙이기, 계산기 함수 연결하기 같은 것부터.

**2단계: 단일 에이전트 완성** — ReAct 루프로 여러 도구를 활용해 복잡한 작업을 처리하는 단일 에이전트를 만든다. "웹 검색해서 요약 보고서 작성"처럼 실제로 쓸만한 에이전트를 완성한다.

**3단계: 멀티에이전트로 확장** — 단일 에이전트가 한계에 부딪히는 지점을 발견하면 그때 분리한다. 처음부터 멀티에이전트를 설계하는 건 오버엔지니어링이다.

### 꼭 기억할 것: Human-in-the-loop

에이전트가 자율적으로 일한다고 해서 인간이 완전히 빠져도 된다는 뜻은 아니다. 특히 초기에는 중요한 행동(파일 삭제, 외부 API 호출, 금전적 거래 등) 전에 반드시 사람의 확인을 받도록 설계해야 한다.

"완전 자동화"는 에이전트가 충분히 신뢰할 수 있다는 확신이 생겼을 때 단계적으로 확대하는 것이다.

## 마무리

AI 에이전트는 "AI가 드디어 실제로 일을 하게 됐다"는 패러다임 전환의 핵심이다. 대화로 답하는 AI에서, 도구를 쓰고 루프를 돌며 목표를 달성하는 AI로.

개발자라면 이 개념들을 지금 이해해두는 게 중요하다. 앞으로 만들게 될 제품, 자동화 파이프라인, 내부 툴들 상당수가 이 에이전트 패턴 위에 지어질 것이기 때문이다.

단순하게 시작하라. 도구 하나 붙이고, 루프 한 번 돌려보고, 실제로 뭔가 일을 시켜봐라. 개념이 이해되는 순간이 반드시 온다.
