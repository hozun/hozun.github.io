## 결론부터

**AI Native는 Cloud Native와 "같은 사고방식, 다른 레이어"다.**

- 둘 다 "사후에 끼워 넣는 것"이 아니라 **처음부터 그 전제 위에서 설계하는 것**을 핵심으로 한다.
- Cloud Native가 인프라/배포 레이어에서의 "클라우드 전제 설계"라면, AI Native는 그 위에서 동작하는 **애플리케이션의 핵심 동작이 AI(주로 LLM)로 구성되는 설계 철학**이다.
- 그래서 AI Native 시스템은 **거의 항상 Cloud Native 위에 얹힌다**. 대체 관계가 아니라 스택 관계다.
- 단, **그대로 베껴 쓰면 깨지는 부분이 명확히 있다.** 결정성, 상태, 비용, 테스트 — 이 네 가지는 Cloud Native의 직관이 그대로 안 통한다.

이 글의 목적: Cloud Native를 어느 정도 이해한 사람이 AI Native라는 단어를 만났을 때, 무엇이 같고 무엇이 새로 배워야 할지 한 번에 매핑하는 것.

---

## 1. Cloud Native 짧게 복습

CNCF의 정의를 거칠게 줄이면 다음과 같다.

> 클라우드 환경(퍼블릭/프라이빗/하이브리드)에서 **확장 가능하고 회복 탄력성 있는 애플리케이션을 만들고 운영할 수 있게 해주는** 기술. 컨테이너, 서비스 메시, 마이크로서비스, 불변 인프라(immutable infra), 선언적 API가 대표 예시.

핵심은 **"클라우드를 단순한 호스팅 장소가 아니라 설계 전제로 삼는다"**는 점이다.

| 영역 | Cloud Native 원칙 |
|------|-------------------|
| 패키징 | 컨테이너 (어디서나 동일하게 실행) |
| 배포 | 선언적(declarative), GitOps |
| 인프라 | 불변(immutable), 일회용(cattle, not pets) |
| 확장 | 수평 확장, 무상태(stateless) 선호 |
| 장애 | "당연히 실패함" 가정 → 재시도, 서킷 브레이커 |
| 운영 | 관찰성(metrics/logs/traces) 기본 |

Cloud Native 개발자가 코드를 짤 때 머릿속에 깔린 가정: **"내 프로세스는 언제든 죽을 수 있고, 똑같은 입력에는 똑같은 출력이 나오며, 트래픽이 늘면 인스턴스를 더 띄우면 된다."**

이 가정 중 일부가 AI Native에서 깨진다. 어디서 깨지는지가 핵심이다.

## 2. AI Native 정의

업계에서 한 줄로 정착된 정의는 아직 없지만, 다음 정도로 모으면 큰 무리가 없다.

> **AI Native 애플리케이션은 핵심 사용자 가치가 AI 모델(주로 LLM)의 추론(inference)에 의해 만들어지는 시스템.** AI는 부가 기능이 아니라 제품의 중심 동작 경로에 있다.

판별 기준 하나: **AI를 빼면 제품이 성립하지 않는가?**

- 검색창 옆에 "AI 요약" 버튼이 있는 종래 SaaS → AI-powered, AI Native 아님
- 사용자가 자연어로 요구하면 에이전트가 도구를 호출해 작업을 끝까지 수행 → AI Native
- 코드 자동완성이 핵심 기능인 IDE 어시스턴트 → AI Native
- 고객 지원 봇이 RAG로 답변하고 필요하면 티켓을 자동 생성/라우팅 → AI Native

### AI Native 스택의 구성 요소

Cloud Native에 "12-factor app" 비슷한 체크리스트가 있듯, AI Native에도 사실상의 표준 컴포넌트가 자리잡고 있다.

```
┌─────────────────────────────────────────────────────┐
│  Eval / Observability (LangSmith, Langfuse, ...)    │  ← AI 전용 관찰성
├─────────────────────────────────────────────────────┤
│  Agent / Orchestration (LangGraph, AgentSDK, MCP)   │  ← 추론 흐름 제어
├─────────────────────────────────────────────────────┤
│  Tools / Function Calling / RAG                     │  ← 외부 세계 연결
├─────────────────────────────────────────────────────┤
│  Context Layer (Memory, Vector DB, Prompt Cache)    │  ← 상태/지식
├─────────────────────────────────────────────────────┤
│  Model Layer (Claude, GPT, Gemini, OSS LLM)         │  ← 추론 엔진
├─────────────────────────────────────────────────────┤
│  Cloud Native Infra (K8s, GPU 노드풀, 서버리스)        │  ← 토대
└─────────────────────────────────────────────────────┘
```

맨 아래가 Cloud Native라는 점에 주목. **AI Native는 Cloud Native를 대체하는 게 아니라 그 위에 두 층(컨텍스트, 에이전트)을 더 올린 것이다.**

## 3. 공통점 — "처음부터 전제로 설계"

| 사고방식 | Cloud Native에서는 | AI Native에서는 |
|----------|----------------------|------------------|
| 핵심 전제 | 클라우드 환경을 가정 | 모델 추론을 가정 |
| 실패 가정 | 노드/프로세스는 죽음 | 모델은 틀린 답을 함 |
| 설계 단위 | 마이크로서비스 | 에이전트/도구 |
| 인터페이스 | API/이벤트 | 자연어 + 도구 호출 |
| 운영 마인드 | 관찰 가능해야 함 | 평가(eval) 가능해야 함 |
| 안티패턴 | "리프트 앤 시프트만 해놓고 끝" | "기존 앱에 챗봇만 붙임" |

특히 안티패턴 행이 비슷하다. Cloud Native 초기에 흔하던 "VM을 컨테이너로 감싸기만 한 가짜 클라우드 네이티브"와, 지금 흔한 "기존 워크플로우 옆에 챗 UI만 붙인 가짜 AI Native"는 같은 종류의 실수다. **밑에 깔린 설계는 그대로면서 표면만 바꾸는 것.**

## 4. 결정적으로 다른 네 가지

여기가 진짜 본론이다. Cloud Native에서 기본기로 굳어진 직관이 AI Native에서는 통하지 않는다.

### 4.1 결정성(Determinism)

Cloud Native:
- 같은 입력 → 같은 출력. 이게 깨지면 버그.
- 캐싱, 멱등성(idempotency) 설계의 전제.

AI Native:
- **같은 입력 → 다른 출력이 정상.** temperature, sampling, 모델 버전, 컨텍스트 윈도우 잘림에 따라 답이 달라진다.
- 그래서 "함수 단위 단위 테스트"가 의미를 잃는다. 대신 **eval set 기반의 통계적 검증**이 필요.
- 캐싱은 가능하지만 "프롬프트 캐시"라는 별도 개념(Anthropic, OpenAI 모두 지원)으로, 비용 절감용이지 결정성 보장용이 아니다.

### 4.2 상태(State)

Cloud Native:
- 무상태(stateless)가 선(善). 상태는 외부 저장소로 빼는 게 정석.
- 인스턴스는 cattle, 언제든 교체 가능.

AI Native:
- **컨텍스트가 곧 상태다.** 그리고 컨텍스트는 토큰 단위로 비용이 매겨진다.
- 대화 히스토리, 메모리, RAG로 검색해 온 문서가 모두 매 호출마다 다시 모델에 들어간다.
- "상태를 외부로 뺀다"는 원칙은 그대로지만, **그 외부 저장소(벡터 DB, 메모리 서비스)가 어떤 토큰을 컨텍스트에 넣을지가 품질을 결정**한다. 단순 보관소가 아니라 검색/요약/우선순위 결정이 들어가는 능동적 컴포넌트.

### 4.3 비용 모델

Cloud Native:
- CPU·메모리·네트워크 시간 단위로 과금. 트래픽이 두 배면 인스턴스를 두 배 띄우면 (대충) 비용도 두 배.
- 캐시 히트율 같은 일반적 최적화가 잘 통함.

AI Native:
- **토큰 단위 과금.** 게다가 입력 토큰과 출력 토큰 단가가 다르고, 모델·캐시 히트 여부에 따라 또 다르다.
- 사용자 한 명의 한 번의 요청이 내부적으로 모델을 5~20번 호출하는 에이전트 워크플로우라면, 호출당 비용 곱하기 호출 횟수가 그대로 단가가 된다.
- "Latency 줄이려고 더 큰 모델 쓰기"가 비용을 몇 배 키울 수 있다. Cloud Native의 "서버 사양 올리기"보다 비용 민감도가 훨씬 가파르다.

### 4.4 테스트와 CI/CD

Cloud Native:
- 단위 테스트 → 통합 테스트 → 배포. 같은 코드면 같은 결과니까 파이프라인이 명확.

AI Native:
- 코드는 그대로인데 **모델 버전이 바뀌면 출력이 바뀐다.** Claude Opus 4.6 → 4.7 같은 마이너 업데이트만으로도 회귀가 가능.
- 그래서 **eval 파이프라인**이 CI에 들어간다. "이 케이스들에 대해 평가 점수가 임계치 이상" 같은 게이트가 단위 테스트 자리를 부분적으로 대체한다.
- 프롬프트 버전 관리, 모델 버전 핀(pin), 카나리 배포(일부 트래픽만 새 모델로) 같은 게 일상.

## 5. Cloud Native 경험이 그대로 쓸모 있는 부분

깨지는 부분만 보면 다 새로 배우는 것 같지만, 그렇지 않다. **운영의 8할은 Cloud Native 그대로다.**

- **컨테이너화**: 에이전트 런타임도 결국 컨테이너에 들어간다.
- **GPU 노드풀, 오토스케일링**: 모델 셀프호스팅하면 K8s + GPU 스케줄링이 그대로 필요.
- **API 게이트웨이, 인증, 레이트 리밋**: LLM API 앞단에도 똑같이 필요. 오히려 토큰 비용 폭주 막으려고 더 엄격하게.
- **관찰성(observability)**: traces/metrics/logs 위에 prompt/completion/token-usage가 한 차원 더 얹힐 뿐 토대는 동일.
- **시크릿 관리, IaC, GitOps**: 모델 API 키, 벡터 DB 자격증명 등 관리 대상만 늘어남.
- **장애 격리·서킷 브레이커**: LLM 호출은 외부 의존성 중에서도 가장 불안정한 축. Cloud Native에서 배운 회복 탄력성 패턴이 더 절실하다.

Cloud Native 백그라운드가 있는 사람이 AI Native 프로젝트에 합류하면, **운영/배포 쪽 토대는 거의 즉시 기여 가능**하다. 새로 배워야 하는 건 위 4가지(결정성/상태/비용/eval)와 그 위의 에이전트 설계 패턴.

## 6. 그래서 어떻게 학습 경로를 잡을까

Cloud Native를 알고 있다는 전제로, 추천 순서.

1. **LLM API 1주일**: Claude 또는 OpenAI API로 간단한 챗 호출, 토큰 단가 직접 체감. 같은 프롬프트 반복 호출해서 출력 분산 확인.
2. **프롬프트 캐싱·구조화 출력**: 비용·결정성 두 문제의 첫 완화책. Cloud Native의 "캐시 레이어 설계"와 비교하면서 보면 빨리 이해됨.
3. **RAG**: 벡터 DB + 검색 + 컨텍스트 주입. "상태를 외부로 빼는" Cloud Native 직관이 가장 잘 맞는 영역.
4. **Function calling / Tool use**: 마이크로서비스 API 호출과 거의 같다고 봐도 됨. 다만 호출 주체가 사람 코드가 아니라 모델.
5. **에이전트와 MCP**: 도구의 표준화·재사용 레이어. 여기서부터 Cloud Native에 정확히 대응하는 개념이 없어서 별도 학습 필요.
6. **Eval**: 마지막에 두지만 사실 가장 중요. 평가 없는 AI Native는 모니터링 없는 Cloud Native와 같다.

---

## 정리

- **Cloud Native** = 클라우드를 전제로 한 인프라/배포 철학.
- **AI Native** = 모델 추론을 전제로 한 애플리케이션 철학. Cloud Native 위에 얹힌다.
- 사고방식("처음부터 X 전제로 설계", "실패가 기본")은 같다. 직관 그대로 통하는 운영·인프라 영역이 8할쯤 된다.
- 단, **결정성 없음 / 컨텍스트가 상태 / 토큰 비용 / eval 기반 검증** — 이 네 가지는 Cloud Native에 대응 개념이 없어서 새로 배워야 한다.
- "AI Native 한다"는 말이 나오면, 진짜 핵심 동작이 AI에 의존하는지 아니면 챗 UI만 붙인 건지부터 구분하는 게 첫 단계다.

Cloud Native 공부한 게 낭비가 아니다. 오히려 토대가 같아서, 그 위에 올라가는 새로운 두 층만 잘 얹으면 된다.
