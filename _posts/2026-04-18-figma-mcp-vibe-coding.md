Figma MCP 얘기가 나올 때마다 살짝 헷갈리는 부분이 있다.

> "AI로 Figma 디자인을 한다는 건가? 아니면 Figma에서 디자인해놓은 걸 AI가 코드로 옮긴다는 건가?"

결론부터: **둘 다 맞다.** 그런데 이 두 가지는 사용 맥락이 완전히 다르고, 지금 시점에서 실제로 쓸 수 있는 건 한쪽에 더 가깝다. 정리해보자.

---

## Figma MCP가 뭔가

Figma가 공식으로 만든 MCP 서버다. 서드파티가 아니라 Figma에서 직접 개발하고 관리한다.

MCP(Model Context Protocol)는 AI 클라이언트가 외부 서비스와 표준화된 방식으로 연결하는 프로토콜이다. Figma MCP 서버를 연결하면, Claude Code·Cursor·Windsurf 같은 AI 개발 도구가 Figma 파일에 직접 접근할 수 있게 된다.

여기서 핵심은 **"직접 접근"**이다. 스크린샷을 찍어서 이미지 분석으로 코드를 생성하는 방식이 아니라, Figma의 **구조화된 데이터 자체**를 읽는다. 컴포넌트 이름, 레이아웃 규칙, 디자인 토큰, 변수값이 전부 텍스트 데이터로 AI에게 전달된다.

---

## 두 가지 방향: 무엇이 다른가

### 방향 1: Figma 디자인 → 코드 (주된 사용 케이스)

가장 많이 쓰이는 방향이다. **디자이너가 Figma에서 화면을 만들고, 개발자가 AI에게 그걸 코드로 옮겨달라는 구조**다.

기존에도 비슷한 시도는 있었다. Figma 화면을 캡처해서 ChatGPT나 Claude에 이미지로 붙여넣으면 대충 비슷한 UI 코드를 뽑아줬다. 그런데 이미지 분석이라 정확도가 낮고, 기존 컴포넌트 라이브러리와 연결이 안 됐다.

Figma MCP는 이걸 다르게 푼다.

```
[Figma 파일]
   ↓ 구조화된 데이터로 읽기
   - Button 컴포넌트 → 내 코드베이스의 <Button /> 연결
   - spacing: 16px → 디자인 토큰 spacing-4 연결
   - color: #0066FF → primary 변수 연결
   ↓
[AI] 이 구조에 맞춰 실제 코드 생성
   ↓
[코드] 기존 컴포넌트와 일관된 결과물
```

**Code Connect**라는 기능도 여기에 연결된다. Figma 컴포넌트와 실제 코드베이스의 컴포넌트를 매핑해두면, AI가 코드를 생성할 때 새로 만들지 않고 기존 컴포넌트를 호출하는 코드를 짠다. 디자인 시스템을 가진 팀이라면 이게 크다.

### 방향 2: AI가 Figma에 직접 디자인을 작성

반대 방향도 가능하다. **AI가 Figma 파일을 읽기만 하는 게 아니라, 직접 Figma에 프레임·컴포넌트·변수를 생성하거나 수정**할 수 있다.

예를 들어:

- "회원가입 플로우 다이어그램을 FigJam에 만들어줘" → AI가 Mermaid 구문을 Figma 다이어그램으로 변환해서 직접 Figma에 그림
- "이 컴포넌트에 다크모드 변수 추가해줘" → AI가 Figma 파일에 변수를 직접 생성
- "프로덕션 코드를 Figma 레이어로 가져와줘" → 반대 방향, 코드에서 Figma로

Figma 블로그에서 소개한 재밌는 케이스 중 하나가 "Claude Code → Figma" 방향이다. 실제로 돌아가는 프로덕션 코드를 Figma가 읽어서 편집 가능한 Figma 디자인으로 역변환하는 것. 디자인 파일 없이 개발된 레거시 코드를 다시 디자인 시스템으로 가져올 때 쓸 수 있다.

---

## "Vibe Coding에서 Figma를 재료로 쓴다"는 게 이런 뜻

Vibe Coding의 맥락에서 Figma MCP를 이야기할 때, 일반적으로 **방향 1 — Figma를 재료로 코드를 만든다**는 쪽이다.

워크플로우를 그려보면:

```
디자이너가 Figma에서 화면 설계
        ↓
개발자(또는 혼자 하는 사람)가 Cursor/Claude Code 열기
        ↓
"이 Figma 프레임을 React 컴포넌트로 만들어줘"
        ↓
AI가 Figma MCP 통해 레이아웃·컴포넌트·색상·여백 데이터 읽기
        ↓
기존 컴포넌트 라이브러리와 연결해서 코드 생성
        ↓
픽셀 수준으로 맞아떨어지는 UI 코드
```

Vibe Coding의 핵심이 "구현 세부사항은 AI에게 맡기고 의도만 전달한다"는 것인데, Figma 파일이 그 "의도"를 가장 구체적이고 구조적으로 담은 문서가 된다. 말로 설명하는 것보다 훨씬 정확하다.

## 실제로 어떻게 연결하나

Figma MCP는 두 가지 서버 옵션을 제공한다.

| 구분 | Remote Server | Local Dev Mode Server |
|------|--------------|----------------------|
| 필요한 것 | Figma 계정 + API 토큰 | Figma 데스크톱 앱 실행 |
| 권장 여부 | 권장 | 제한적 |
| 특징 | 더 많은 기능, 앱 불필요 | 로컬에서만 동작 |

Claude Code에서 연결하는 방법:

```bash
claude mcp add figma
```

또는 `settings.json`에 직접 추가:

```json
{
  "mcpServers": {
    "figma": {
      "command": "npx",
      "args": ["-y", "figma-mcp"]
    }
  }
}
```

Cursor는 Settings → MCP 섹션에서 SSE 방식으로 연결한다.

## 현재 상태와 한계

**무료 베타 중**이다. 향후 사용량 기반 유료 전환 예정.

몇 가지 현실적인 한계:

**디자인 파일 품질에 의존한다** — Figma 파일이 컴포넌트와 변수를 잘 정리해놓은 상태여야 AI가 제대로 읽는다. 레이어 이름이 "Frame 123", "Group 45" 식으로 아무렇게나 된 파일이면 결과도 엉망이다.

**Code Connect 설정이 선행되어야 한다** — Figma 컴포넌트와 코드 컴포넌트를 연결하는 작업을 미리 해둬야 기존 라이브러리와 연동된 코드가 나온다. 이 설정 없이 쓰면 AI가 컴포넌트를 새로 만들어버린다.

**복잡한 인터랙션은 아직** — 정적인 레이아웃과 스타일은 잘 옮기는데, 애니메이션이나 복잡한 상태 관리는 별도로 지시해야 한다.

## 정리

| 질문 | 답 |
|------|---|
| AI가 Figma 디자인 자체를 하는 건가? | 가능하다. 하지만 보조적 역할 (다이어그램 생성, 변수 추가 등) |
| Figma 디자인을 코드의 재료로 쓰는 건가? | 이게 핵심 사용 케이스다 |
| Vibe Coding에서의 역할은? | Figma가 "의도를 담은 구조화된 설계서"로 AI에게 전달되는 재료 |
| 지금 당장 쓸 수 있나? | 무료 베타. Figma 파일이 잘 정리되어 있으면 바로 써볼 만하다 |

결국 Figma MCP의 가장 강력한 포인트는 **디자인-개발 간 번역 비용을 줄이는 것**이다. 디자이너가 의도한 것과 개발자가 구현한 것 사이의 간극, 그 지점을 AI가 메운다. 디자인 시스템이 잘 갖춰진 팀이라면 지금 당장 도입해볼 가치가 있다.
