앞선 [SDD 확산 글](/2026/07/22/sdd-diffusion-spec-as-contract.html)에서 "스펙이 계약이 된다"고 썼다. 그런데 대부분의 SDD 논의는 **AI가 코드까지 짜는 것**을 전제한다. 여기서 방향을 하나 틀어보자.

> **AI는 스펙과 태스크까지만 만들고, 실제 구현은 사람이 로우코드 플랫폼에서 한다.**

이게 왜 흥미로운가. 사람이 각 태스크를 손으로 조립하는 과정에서 **휴먼인더루프와 산출물 점검이 공짜로 따라온다.** 속도는 AI 풀 코드젠보다 느리지만, 끝에는 **사람이 이해하고 소유한, 실제로 동작하는 서비스**가 남는다. 로우코드는 해법 공간이 좁아서(엔티티·스크린·액션 몇 종류) 스펙에서 구현으로의 번역 손실도 작다. SDD와 궁합이 좋을 수밖에 없다.

말로만 하면 와닿지 않으니, **산출물을 실제로 펼쳐** 보이겠다. 소재는 어느 회사에나 있는 **연차 휴가 신청·결재 앱**이다.

---

## 전체 흐름: 어디까지 AI, 어디부터 사람

```
 사람(현업/기획)        AI 에이전트              사람(로우코드 개발자)
 ─────────────         ────────────            ────────────────────
   요구사항       ──►    스펙 생성       ──►         태스크를
   (자연어)              (구조화)                    하나씩 구현
      │                    │                    (Entity/Screen/Action)
      │                    ▼                           │
      └──── 리뷰 ◄──── 태스크 분해 ──► 리뷰 ─────────►  │
                       (빌드 순서표)                    ▼
                                                  동작하는 서비스
                                                  + 각 단계 점검 완료
```

핵심은 화살표 사이사이의 **리뷰**다. 요구사항→스펙에서 한 번, 스펙→태스크에서 한 번, 그리고 사람이 각 태스크를 구현하며 매번. AI가 코드까지 다 짜버리면 이 점검 지점들이 뭉개진다. 사람이 짓기 때문에 점검이 강제된다.

---

## 산출물 1 — 요구사항 (사람이 입력)

먼저 **사람이 넣는 입력의 수준**부터. 흔한 실패는 이렇게 던지는 것이다.

```
"휴가 신청하고 팀장이 결재하는 앱 만들어줘"
```

이 한 줄로는 AI가 잔여일수 처리, 반차 규칙, 알림 여부를 전부 추측한다. 추측이 곧 재작업이다.

반대로, 아래 정도면 충분하다. **전문 용어도, 데이터 설계도 필요 없다.** 현업 담당자가 아는 만큼만 구조 있게 적으면 된다.

```markdown
## 만들려는 것
직원이 연차/반차를 신청하고 팀장이 승인·반려하는 사내 앱.
지금은 엑셀+메일로 처리하는데, 잔여일수 계산이 자주 틀리고
누가 언제 썼는지 이력 추적이 안 된다.

## 사용자
- 직원: 휴가 신청, 내 신청 현황·잔여일수 조회
- 팀장: 팀원 신청 건 승인/반려 (반려 시 사유 필수)
- 인사담당자: 전사 휴가 현황 조회, 연차 부여·조정

## 꼭 지켜야 할 규칙
- 잔여일수보다 많이 신청할 수 없다
- 반차는 0.5일, 주말·공휴일은 사용 일수에서 제외
- 결재는 1단계(직속 팀장)만
- 승인/반려되면 신청자에게 알림
- 반려된 건은 수정해서 재신청 가능
```

포인트: **"무엇을, 누가, 어떤 규칙으로"**만 담으면 된다. "Entity를 어떻게 설계하라"는 여기 없다. 그건 AI가 스펙에서 채운다.

## 산출물 2 — 스펙 (AI 생성)

위 요구사항을 넣으면 AI가 아래 같은 스펙을 만든다. **모호했던 것들이 구조로 확정**된다.

### 2-1. 데이터 모델

요구사항엔 "직원", "휴가", "잔여일수"라는 단어만 있었는데, AI가 엔티티와 속성, 관계로 확정한다.

```
┌──────────────┐        ┌────────────────────┐
│  Employee    │ 1    N │  LeaveRequest      │
│──────────────│◄───────│────────────────────│
│ Id           │        │ Id                 │
│ Name         │        │ EmployeeId (FK)    │
│ ManagerId(FK)│──┐     │ Type (연차/반차)   │
│ RoleId  (FK) │  │자기 │ StartDate          │
└──────────────┘  │참조 │ EndDate            │
       │ 1        └─────►│ UsedDays (계산값)  │
       │                 │ Status             │
       │ N               │ RejectReason       │
┌──────────────┐         │ CreatedOn          │
│ LeaveBalance │         └────────────────────┘
│──────────────│
│ EmployeeId FK│         ┌──────────────┐
│ Year         │         │ Holiday      │  (공휴일 참조)
│ GrantedDays  │         │──────────────│
│ UsedDays     │         │ Date         │
│ RemainingDays│         │ Name         │
└──────────────┘         └──────────────┘
```

| 엔티티 | 역할 | 핵심 속성 |
|---|---|---|
| **Employee** | 직원·조직 | Name, ManagerId(직속 팀장), RoleId |
| **LeaveRequest** | 휴가 신청 건 | Type, StartDate, EndDate, UsedDays, Status(대기/승인/반려), RejectReason |
| **LeaveBalance** | 연도별 잔여 | GrantedDays, UsedDays, RemainingDays |
| **Holiday** | 공휴일 참조 데이터 | Date, Name |

### 2-2. 역할과 권한

| Role | 볼 수 있는 것 | 할 수 있는 것 |
|---|---|---|
| Employee | 내 신청·잔여 | 신청, 재신청 |
| Manager | 내 팀원 신청 | 승인/반려 (Employee 권한 포함) |
| HR | 전사 현황 | 연차 부여·조정 |

### 2-3. 사용자 흐름 (신청 → 결재)

```
직원                시스템                 팀장
 │  신청 작성         │                      │
 │───────────────────►│ 잔여일수 검증        │
 │                    │ (부족하면 거절)      │
 │                    │ Status=대기          │
 │                    │────── 알림 ─────────►│
 │                    │                      │ 결재함에서 확인
 │                    │◄──── 승인/반려 ──────│ (반려 시 사유 필수)
 │◄───── 알림 ────────│ 잔여 차감(승인 시)   │
 │                    │ Status=승인/반려     │
```

### 2-4. 수용 기준 (EARS 표기)

여기가 스펙의 심장이다. **검증 가능한 문장**으로 규칙을 못 박는다. 뒤의 태스크와 테스트가 전부 이 항목을 가리킨다.

```
EARS-1  WHEN 직원이 신청을 제출할 때,
        IF 신청 일수 > 잔여 일수 이면,
        시스템은 SHALL 제출을 막고 "잔여일수 부족" 메시지를 보여준다.

EARS-2  WHEN 팀장이 신청 건을 반려할 때,
        IF 반려 사유가 비어 있으면,
        시스템은 SHALL 반려를 막고 사유 입력을 요구한다.

EARS-3  WHEN 신청 일수를 계산할 때,
        시스템은 SHALL 시작일~종료일 중 주말과 Holiday를 제외한다.
        (반차는 0.5일로 계산한다)

EARS-4  WHEN 신청이 승인될 때,
        시스템은 SHALL 해당 직원의 LeaveBalance.UsedDays를 증가시키고
        RemainingDays를 재계산한다.

EARS-5  WHEN 신청이 승인 또는 반려될 때,
        시스템은 SHALL 신청자에게 인앱 알림과 이메일을 발송한다.
```

### 2-5. 제약 (플랫폼)

```
- 플랫폼: OutSystems (Mendix / Power Apps에도 동일 구조로 매핑됨)
- 결재는 1단계만. 다단계·대결·위임은 이번 범위 아님
- 외부 캘린더/그룹웨어 연동 없음. Holiday는 수동 부트스트랩
- 알림: 인앱 + 이메일. 카카오/SMS 아웃 오브 스코프
```

이 "**하지 않을 것**" 목록이 중요하다. 로우코드는 이것저것 붙이기 쉬워서, 범위를 안 막으면 사람이 스스로 스코프를 부풀린다.

## 산출물 3 — 태스크 (AI 생성, 사람에게 전달)

이제 이 방식의 진짜 차별점. 보통 SDD에서 태스크는 **AI 코딩 에이전트**에게 준다. 여기서는 **로우코드로 짓는 사람**에게 준다. 그래서 태스크의 언어가 "함수를 짜라"가 아니라 **"플랫폼의 이 요소를 이렇게 만들라 + 다 됐는지 이렇게 확인하라"**가 된다.

각 태스크 카드는 네 칸이다: **목적 / 플랫폼 작업 / 스펙 추적 / 완료(리뷰) 조건.**

```markdown
── T1. 데이터 모델 생성 ─────────────────────────────
목적      휴가 신청을 저장할 구조를 만든다
플랫폼작업 Entity 4개 생성: Employee, LeaveRequest,
          LeaveBalance, Holiday. 위 속성표대로.
          Employee.ManagerId는 Employee 자기참조 FK.
스펙추적   §2-1 데이터 모델
완료조건   ✅ 4개 Entity가 Publish됨
          ✅ ManagerId로 팀장-팀원 조회가 되는지 샘플
             데이터 2건으로 확인
```

```markdown
── T3. 신청 일수 계산 로직 ───────────────────────────
목적      주말·공휴일 빼고 실제 사용 일수를 구한다
플랫폼작업 Server Action: CalculateLeaveDays
          IN  StartDate, EndDate, IsHalfDay
          OUT UsedDays
          로직: 날짜 순회하며 주말/Holiday 제외,
               반차면 0.5 반환
스펙추적   EARS-3
완료조건   ✅ 금~월(주말 포함) 신청 → 2일로 계산
          ✅ 광복절 낀 구간 → 그날 제외되는지 확인
          ✅ 반차 → 0.5 반환
```

```markdown
── T6. 팀장 결재함 + 승인/반려 ───────────────────────
목적      팀장이 팀원 신청을 처리한다
플랫폼작업 Screen: ApprovalInbox (Role=Manager 접근)
          Aggregate: Status=대기 AND
                     Employee.ManagerId=현재사용자
          Action: Approve / Reject
          Reject는 사유 필수(Form 검증)
스펙추적   EARS-2, §2-3 흐름
완료조건   ✅ 다른 팀 신청 건은 안 보임
          ✅ 사유 없이 반려 시도 → 막힘
          ✅ 승인 시 T7 알림이 트리거됨
```

나머지 태스크는 같은 형식으로 이어진다 (요약).

| 태스크 | 플랫폼 산출물 | 스펙 추적 |
|---|---|---|
| T2 | Holiday 부트스트랩 + Role 3종 정의 | §2-2, §2-5 |
| T4 | 휴가 신청 Screen + Form (잔여 검증) | EARS-1 |
| T5 | 내 신청 현황 Screen (본인 필터 Aggregate) | §2-2 |
| T7 | 승인/반려 알림 (인앱 + 이메일) | EARS-5 |
| T8 | 인사 대시보드 Screen (Role=HR) | §2-2 |
| T9 | 승인 시 잔여 차감 로직 | EARS-4 |

각 태스크가 **작아서 리뷰 단위**가 되고, **완료 조건이 곧 테스트 시나리오**다. 사람이 T3을 만들고 "광복절 낀 구간" 하나 눌러보는 순간, 그게 휴먼인더루프다.

## 추적성: 이게 있어야 점검이 된다

세 산출물을 관통하는 한 줄짜리 표. 요구사항 하나가 스펙을 거쳐 태스크까지 어떻게 내려오는지 보인다.

| 요구사항 | 스펙 조항 | 태스크 | 확인 방법 |
|---|---|---|---|
| 잔여보다 많이 신청 불가 | EARS-1 | T4 | 잔여 1일에 3일 신청 → 막힘 |
| 반차 0.5, 주말·공휴일 제외 | EARS-3 | T3 | 반차·공휴일 케이스 계산 |
| 반려 사유 필수 | EARS-2 | T6 | 빈 사유로 반려 → 막힘 |
| 승인 시 잔여 차감 | EARS-4 | T9 | 승인 후 잔여 감소 확인 |
| 승인·반려 시 알림 | EARS-5 | T7 | 알림/메일 수신 확인 |

거꾸로도 성립한다. 나중에 버그가 나면 "어느 태스크 → 어느 스펙 조항 → 원래 어떤 요구사항"인지 역추적된다. AI가 코드를 통째로 뱉었다면 이 사슬이 없다.

## 그래서, 이 방식이 로우코드에서 효율적인가

내 결론은 **"그렇다, 단 조건부로"**다. 냉정하게 갈라 보면.

**잘 통하는 이유**

- **번역 손실이 작다.** 로우코드의 구성 요소는 Entity·Screen·Action·Aggregate 정도로 유한하다. 스펙의 "데이터 모델→Entity", "흐름→Screen/Action" 매핑이 거의 1:1이라, 자유도가 무한한 코드보다 스펙과 구현의 거리가 짧다.
- **태스크가 곧 빌드 순서표.** 로우코드 개발자의 흔한 마비는 "다음에 뭘 만들지"다. 태스크 목록이 그걸 없앤다.
- **점검이 공짜.** 사람이 각 요소를 손으로 만들며 완료 조건을 눌러보므로, 별도 QA 단계 없이 산출물이 검증된다.
- **결과물을 사람이 이해·소유한다.** AI가 만든 블랙박스가 아니라, 만든 사람이 구조를 안다. 유지보수·인수인계가 된다.

**한계 / 안 통하는 곳**

- **느리다.** 순수 CRUD 화면 20개처럼 반복이 많은 작업은, 사람이 클릭으로 짓는 것보다 AI 코드젠(또는 로우코드 자체 AI)이 빠르다. 이 방식의 가치는 "속도"가 아니라 "통제·이해"에 있다.
- **스펙 품질에 전부 걸려 있다.** 요구사항·스펙이 엉성하면 태스크도 엉성하고, 사람은 잘못된 순서표를 성실히 따를 뿐이다. 앞단 리뷰가 무너지면 전체가 무너진다.
- **큰 앞단 설계의 위험.** 지난 글에서 짚은 회의론 — 앞에 스펙을 과설계하면 반복적 발견이 줄어든다 — 이 여기도 적용된다. 그래서 **한 번에 앱 전체 스펙이 아니라, 기능 단위로 요구사항→스펙→태스크를 짧게 돌리는 것**이 맞다.

정리하면, 사용자가 구상한 그림 — "생성된 스펙·태스크로 사람이 로우코드를 짓고, 그 과정에서 휴먼인더루프와 점검이 자연히 일어나 실제 동작하는 서비스가 완성된다" — 은 **성립한다.** 다만 그 강점은 시간 단축이 아니라 **통제·추적·이해의 보존**에 있고, 반복 노가다형 화면에는 굳이 이 방식을 고집할 필요가 없다.

## 정리

```
요구사항 (사람)  →  "무엇을·누가·어떤 규칙으로"만, 자연어로
   스펙 (AI)     →  데이터 모델 + 역할 + 흐름 + EARS 수용기준 + 제약
  태스크 (AI)    →  "이 요소를 이렇게 만들고, 이렇게 확인하라"
  구현 (사람)    →  로우코드로 하나씩. 완료 조건 = 리뷰 게이트
```

AI에게 코드까지 다 맡기는 것과, 아무 지도 없이 로우코드를 헤매는 것. 그 사이에 **"AI가 스펙·태스크를, 사람이 구현을"** 나눠 갖는 지대가 있다. 로우코드는 해법 공간이 좁아서 이 분업이 특히 깔끔하게 떨어진다.

느리지만 사람이 만든다 — 그래서 완성되면 그건 **점검이 끝난, 사람이 소유한 진짜 서비스**다. 다음 글에서는 이 워크플로우를 spec-kit이나 Kiro 같은 실제 툴 출력과 붙여, 위 산출물을 자동 생성해 보는 실습을 다뤄볼 생각이다.
