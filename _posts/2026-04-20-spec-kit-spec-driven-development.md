AI 코딩 도구를 써본 경험이 어느 정도 있다면 한 번쯤 이런 상황을 겪었을 것이다.

"처음엔 잘 됐는데, 기능이 몇 개 붙고 나서부터 AI가 이상한 방향으로 가기 시작했다."

AI한테 뭔가를 시키면 답은 나오는데, 전체 구조와 맞지 않는 코드가 나온다. 앞에서 정한 패턴을 무시하고 다른 방식으로 짠다. 맥락이 길어질수록 일관성이 떨어진다.

이 문제를 해결하려는 접근 중 하나가 **Spec-Driven Development**고, GitHub이 이를 체계화한 툴킷이 **spec-kit**이다.

---

## Vibe Coding의 한계

Vibe Coding은 빠르다. 아이디어를 말로 던지면 AI가 코드를 뽑아낸다. 프로토타입 수준에선 강력하다.

문제는 규모다. 프로젝트가 커질수록 AI는 전체 맥락을 유지하기 어렵다. "이전에 우리가 이런 방식으로 하기로 했잖아"를 AI는 기억하지 못한다. 매번 다시 설명해야 하고, 그 설명이 조금씩 달라지면 결과물도 달라진다.

Vibe Coding의 문제는 **"말"이 사라진다는 것**이다. 대화창에 흘러간 지시사항은 컨텍스트 밖으로 밀려나고, 프로젝트 어딘가에 남지 않는다.

---

## Spec-Driven Development: 먼저 문서, 그 다음 코드

Spec-Driven Development의 핵심은 단순하다.

**AI에게 코드를 시키기 전에, 먼저 사양(spec)을 문서로 만들어라.**

이 사양이 대화창 안에 있는 게 아니라 **파일로 존재**한다는 게 중요하다. AI는 매 요청마다 이 파일을 참조한다. 구조, 제약사항, 원칙이 문서에 박혀있으니 AI가 엉뚱한 방향으로 가기 어렵다.

생각해보면 이건 사람끼리 협업하는 방식과 같다. 기획서, 아키텍처 문서, 코딩 컨벤션이 있으면 새 팀원도 방향을 잃지 않는다. AI도 마찬가지다.

## spec-kit 구조

GitHub이 만든 오픈소스 CLI 툴킷으로, MIT 라이선스다. Claude Code, GitHub Copilot, Gemini CLI 등 30개 이상의 AI 코딩 도구와 호환된다.

핵심 개념은 4단계 프로세스다.

### 1단계: Specify — 목표 정의

무엇을 만드는지, 누구를 위한 건지, 어떤 사용자 여정이 있는지를 문서화한다.

```markdown
# Project: TaskFlow

## Goal
소규모 팀(5인 이하)을 위한 태스크 관리 앱.
Jira는 무겁고 Notion은 느슨하다. 그 사이 어딘가.

## User Journeys
1. 팀원이 태스크를 생성하고 담당자를 지정한다
2. 담당자는 상태를 업데이트한다
3. 팀 리더는 전체 진행 현황을 한 눈에 본다
```

막연하게 "태스크 관리 앱 만들어줘"가 아니라, AI가 판단할 수 있는 근거를 만드는 단계다.

### 2단계: Plan — 아키텍처와 제약사항

기술 스택, 폴더 구조, 하지 않을 것들을 선언한다. 이 문서가 있으면 AI가 갑자기 다른 라이브러리를 끌어오거나, 정해둔 패턴을 무시하는 상황이 줄어든다.

```markdown
## Stack
- Frontend: Next.js 15, TypeScript
- Styling: Tailwind CSS (CSS-in-JS 사용 안 함)
- State: Zustand (Redux 사용 안 함)
- DB: Supabase

## Constraints
- 서드파티 UI 라이브러리 추가 금지 (shadcn/ui만 허용)
- 파일 하나에 컴포넌트 하나
- any 타입 사용 금지
```

spec-kit에는 **constitution.md**라는 개념이 있다. 헌법처럼 프로젝트에서 절대 어기면 안 되는 원칙을 담는 파일이다. AI는 이 파일을 먼저 참조해서 기본 방향을 잡는다.

### 3단계: Tasks — 작업 단위 분해

큰 목표를 AI가 한 번에 처리할 수 있는 작은 단위로 쪼갠다.

```markdown
## Tasks

- [ ] Task 1: Supabase 스키마 설계 (users, tasks, teams 테이블)
- [ ] Task 2: 인증 플로우 구현 (소셜 로그인만)
- [ ] Task 3: 태스크 생성 API 엔드포인트
- [ ] Task 4: 태스크 목록 UI 컴포넌트
- [ ] Task 5: 상태 변경 기능
```

이렇게 쪼개면 두 가지 장점이 있다. 첫째, AI가 한 번에 처리하는 범위가 좁아서 결과물 품질이 높아진다. 둘째, 각 태스크가 리뷰 단위가 돼서 문제가 생겼을 때 어디서 틀어졌는지 추적하기 쉽다.

### 4단계: Implement — AI가 구현

사양 문서를 AI에게 건네고 구현을 시킨다. AI는 constitution.md → plan → tasks 순서로 맥락을 쌓고 코드를 생성한다.

## 설치와 시작

Python 환경이 있으면 uv로 바로 실행할 수 있다.

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init MY_PROJECT
```

이 명령어 하나로 프로젝트 폴더 구조와 기본 사양 파일들이 생성된다. 이후 각 파일을 채워나가는 방식.

Jira를 쓰는 팀이라면 태스크 문서에서 Jira 에픽·스토리·이슈를 자동 생성하는 기능도 있다.

## "그냥 CLAUDE.md 잘 쓰면 되는 거 아닌가?"

비슷하게 보이지만 초점이 다르다.

| | CLAUDE.md / 시스템 프롬프트 | spec-kit |
|---|---|---|
| 목적 | AI 동작 방식 설정 | 프로젝트 사양 문서화 |
| 내용 | "이렇게 답해라" | "이걸 만들어라, 이 원칙으로" |
| 단위 | 프로젝트 전체 설정 | 기능·태스크 단위 |
| 재사용 | 프로젝트마다 다시 씀 | 사양 → 구현 흐름 자체가 재사용 가능 |

CLAUDE.md는 AI의 **행동 방식**을 설정하는 것이고, spec-kit은 **무엇을 어떻게 만들지**를 구조화하는 것이다. 같이 쓰는 게 맞다.

---

## 어떤 상황에서 쓰면 좋은가

**맞는 상황:**
- 프로토타입을 벗어나서 실제 제품을 AI와 함께 만들 때
- 팀 단위로 AI 개발을 하는데 일관성을 유지하고 싶을 때
- 스택과 컨벤션이 정해진 프로젝트에서 AI를 활용할 때

**덜 맞는 상황:**
- 아이디어를 빠르게 검증하는 1-2일짜리 해킹 프로젝트
- 사양 자체가 아직 안 잡힌 초기 탐색 단계

---

## 정리

spec-kit의 핵심 메시지는 결국 이거다.

> "AI에게 더 잘 시키고 싶으면, 먼저 더 잘 설명해라. 그 설명이 파일로 존재해야 한다."

Vibe Coding이 AI와의 협업을 가능하게 했다면, Spec-Driven Development는 그 협업을 **지속 가능하게** 만드는 방식이다. 두 접근이 대립하는 게 아니라, 프로젝트 규모와 단계에 따라 선택하거나 섞어 쓰는 것이다.

AI 코딩 도구를 쓰면서 "왜 갈수록 통제가 안 되지?"라는 느낌이 들었다면, spec-kit이 그 불편함의 원인과 해결책을 동시에 보여줄 것이다.
