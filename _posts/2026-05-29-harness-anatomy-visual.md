[지난 글](/2026/05/29/harness-engineering/)에서 하네스가 뭔지, 왜 중요한지 정리했다. 이번 글은 "그래서 하네스가 도대체 안에서 뭘 한다는 거냐"에 대한 답이다. 그림으로 보자.

---

## 1. 한 장으로 본 하네스 구조

먼저 전체 구조부터. 모델은 그냥 토큰을 뱉는 함수다. 그 함수 호출을 둘러싼 모든 게 하네스다.

```
[사용자 메시지]
      │
      ▼
[1] 컨텍스트 조립기   ◄── 시스템 프롬프트 + 툴 스키마 + 히스토리
      │
      ▼
[2] 모델 호출         ◄── 외부 LLM API
      │
      ▼
[3] 응답 파서
      │
      ├──► (텍스트만)  ──────────────────►  사용자에게 표시
      │
      └──► (툴 호출)
                │
                ▼
            [4] 권한 게이트  ◄── 자동 승인 / 사용자 확인 정책
                │
                ▼
            [5] 툴 실행기
                │
                ▼
            [6] 결과를 컨텍스트에 추가
                │
                └─────► 루프: 다시 [2] 모델 호출
```

핵심은 **점선 화살표가 아니라 굵은 루프**다. 모델 호출은 한 번이 아니라 여러 번이고, 그 사이를 잇는 컨텍스트·툴·권한·종료조건이 전부 하네스의 일이다.

---

## 2. 한 턴의 라이프사이클

사용자가 "auth 모듈의 deprecated 함수들 정리해줘"라고 입력했다 치자. 하네스 안에서 시간 순서대로 무슨 일이 일어나는지 본다.

```
T+0.0s  사용자: "auth 모듈의 deprecated 함수들 정리해줘"
        │
        ▼
T+0.0s  [하네스] 컨텍스트 조립
        ├─ 시스템 프롬프트 (3.2K tok)
        ├─ 툴 스키마 12개 (1.8K tok)
        ├─ 작업 디렉토리·git status (0.4K tok)
        ├─ 이전 대화 (현재 비어있음)
        └─ 신규 메시지 (0.05K tok)
        총 5.45K tok 전송
        │
        ▼
T+1.2s  [모델] 응답:
        "auth 모듈을 먼저 찾아볼게요."
        + tool_call: Glob(pattern="**/auth/**/*.py")
        │
        ▼
T+1.2s  [하네스] 응답 파서 → 툴 호출 감지
        [하네스] 권한 게이트 → Glob은 read-only, 자동 승인
        [하네스] 툴 실행기 → Glob 호출
        │
        ▼
T+1.3s  [툴 결과] 17개 파일 경로 반환
        [하네스] 결과를 컨텍스트에 추가 (0.3K tok)
        [하네스] 루프 계속, 다시 모델 호출 (총 5.75K tok)
        │
        ▼
T+2.8s  [모델] 응답:
        + tool_call: Grep(pattern="@deprecated", path="src/auth")
        │
        ▼
T+2.9s  [툴 결과] 23개 매칭, 8개 파일
        [하네스] 컨텍스트 추가 (1.1K tok)
        │
        ▼
T+4.4s  [모델] "8개 파일을 읽어 의존성을 확인하겠습니다"
        + tool_call: Read(...)  × 8개 병렬
        │
        ▼
        ... (계속 루프)
        │
T+47s   [모델] 텍스트만 반환 (툴 호출 없음)
        "정리가 끝났습니다. 총 12개 함수를 제거했고..."
        │
        ▼
T+47s   [하네스] 툴 호출 없음 → 종료 조건 충족 → 루프 탈출
        [하네스] 사용자에게 최종 메시지 표시
```

여기서 보이는 것들:

- **모델 호출은 1회가 아니다.** 이 예시에서만 7~10번 정도 호출된다.
- **컨텍스트는 매 호출마다 새로 조립된다.** 매번 시스템 프롬프트·툴 스키마·누적 히스토리를 전부 보낸다. 비용도, 지연도 누적된다.
- **종료는 "모델이 텍스트만 반환"하는 시점에 자동 결정된다.** 모델이 "다 됐다"고 말하는 게 아니라, 툴 호출을 더 안 하는 행동으로 끝낸다.
- **권한 게이트가 매 툴마다 작동한다.** Glob/Read 같은 읽기는 자동, Edit/Bash 같은 쓰기는 정책 따라 사람한테 묻거나 통과.

## 3. 시스템 프롬프트의 단면

"시스템 프롬프트가 뭐 들어가냐"는 자주 가려져 있다. 실제로는 이런 층들로 쌓여 있다.

| # | 레이어 | 예시 내용 | 토큰(대략) |
|---|---|---|---|
| 1 | 정체성·페르소나 | "당신은 Claude Code, 시니어 엔지니어…" | ~200 |
| 2 | 안전·거절 정책 | "악성 코드 작성 거부, 자격증명 노출 금지…" | ~300 |
| 3 | 행동 규칙 | "병렬 도구 호출 우선, 추측 금지, 변경 최소화…" | ~800 |
| 4 | 도구 사용 가이드 | "Read 전에 Glob, 검색은 Grep, 수정은 Edit" | ~500 |
| 5 | 출력 형식 규칙 | "한 문장 업데이트, 마크다운, 코드블록 사용" | ~300 |
| 6 | 환경 컨텍스트 (동적) | OS, Shell, 작업 디렉토리, git branch 등 | ~400 |
| 7 | 프로젝트 규칙 (CLAUDE.md) | "_posts에 글 추가 후 자동 푸시, 태그는…" | ~600 |
| | **합계** | | **~3.1K** |

여기 들어가는 한 줄 한 줄이 모델 행동에 영향을 준다. Layer 3의 "병렬 도구 호출 우선" 한 줄이 빠지면 모델이 도구를 순차로만 호출해서 작업 시간이 2~3배가 된다. Layer 7의 프로젝트 규칙이 없으면 매번 사용자가 같은 잔소리를 반복해야 한다.

**하네스 엔지니어가 만지는 1차 산출물이 바로 이 레이어 구성이다.**

## 4. 컨텍스트 윈도우는 살아 움직인다

또 하나 시각화하기 좋은 게 컨텍스트의 시간축이다. 작업이 길어질수록 컨텍스트가 어떻게 변하는지 보자.

```
턴 1 (5K tok)
████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ (200K 윈도우)
└시스템+툴+첫 메시지

턴 5 (38K tok)
████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░
└시스템+툴+5턴치 대화·툴 결과 누적

턴 12 (94K tok)
████████████████████████████░░░░░░░░░░░░
└여전히 누적 중. Read 결과가 컨텍스트의 60% 차지

턴 18 (152K tok) ⚠ 임계점 근접
████████████████████████████████████░░░░
└하네스가 압축(compaction) 트리거

턴 19 (44K tok) ← 압축 직후
████████░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░
└오래된 턴들을 요약본으로 치환
└중요 사실은 메모리 파일로 빼냄
└현재 작업 컨텍스트는 보존
```

이 압축 정책 자체가 하네스의 한 부분이다.

- 무엇을 요약하고 무엇을 통째로 유지할지
- 어떤 정보를 영구 메모리 파일로 빼낼지
- 압축을 언제 트리거할지 (토큰 임계점? 턴 수? 사용자 명시 요청?)
- 압축본의 형식을 어떻게 만들지 (요약 한 문단? 구조화 JSON?)

같은 모델이라도 이 정책이 다르면 장시간 작업에서 완전히 다른 결과가 나온다. 한쪽은 "초반에 결정한 설계 원칙"을 끝까지 기억하고, 한쪽은 30턴 지나면 잊어버린다.

## 5. 좋은 하네스 vs 나쁜 하네스

같은 모델, 같은 작업, 다른 하네스. 실제로 결과 차이가 어디서 나는지 비교해본다.

### 케이스: "이 함수에 단위 테스트 추가해줘"

**나쁜 하네스의 실행 흐름:**
```
사용자 → 모델
        ├ Read(file.py) [전체 파일 1500줄 로드]
        ├ Read(file.py) [방금 읽은 파일 또 읽음 — 기억 못함]
        ├ Write(test_file.py) [기존 테스트 파일 통째 덮어씀 ⚠]
        ├ Bash("pytest") [전체 테스트 스위트 5분간 실행]
        └ "완료했습니다"

문제:
- 중복 Read (컨텍스트 낭비)
- Write로 통째 덮어씀 (기존 테스트 소실)
- 좁게 실행 안 함 (Bash 전체)
- 검증 없이 완료 선언
```

**좋은 하네스의 실행 흐름:**
```
사용자 → 모델
        ├ Glob("**/test_*.py") [관련 테스트 파일 위치 먼저 확인]
        ├ Read(file.py, lines=120-180) [필요한 함수 부분만]
        ├ Read(test_target.py) [기존 테스트 구조 파악]
        ├ Edit(test_target.py, ...) [기존 파일에 케이스만 추가]
        ├ Bash("pytest test_target.py::TestX -v") [좁은 범위만]
        ├ [실패 1건 감지]
        ├ Edit(test_target.py, ...) [수정]
        ├ Bash("pytest test_target.py::TestX -v") [다시]
        ├ [통과 확인]
        └ "추가한 케이스 3개, 모두 통과"

차이를 만든 하네스 요소:
- 툴 셋에 Glob/Edit가 있고 Write는 신규파일 한정
- 시스템 프롬프트에 "Read 전에 Glob, 좁은 범위로 실행" 규칙
- Bash 결과를 모델이 다시 보고 판단하는 루프
- 종료 전 검증 통과 의무
```

모델이 똑똑해서가 아니다. **하네스가 모델한테 "이런 식으로 일해라"고 강제하고 있는 것**이다. 같은 모델이 나쁜 하네스 안에 들어가면 위의 나쁜 흐름을 그대로 한다.

## 6. 그래서 하네스 엔지니어는 뭘 보고 일하나

마지막으로, 실제 하네스 엔지니어가 화면에 띄워두고 일하는 것들.

**트레이스 뷰어 (한 작업의 턴별 흐름):**

```
턴 1 ──────────────► 의도 해석 / 계획 수립
턴 2 ──────────────► 파일 검색 (Glob)
턴 3 ──[툴 실패]───► 에러 메시지 분석
턴 4 ──[복구 시도]─► 대안 경로로 재시도
턴 5 ──[성공]──────► 결과 보고 / 종료
```

**평가 대시보드 (어제 하네스 vs 오늘 하네스 회귀 비교):**

| 지표 | 어제 | 오늘 | 변화 |
|---|---|---|---|
| 통과율 (pass rate) | 85% | 87% | ▲ |
| 평균 턴 수 | 7.2 | 6.4 | ▼ (좋음) |
| 평균 비용 | $0.21 | $0.18 | ▼ (좋음) |
| p95 지연 | 51s | 47s | ▼ (좋음) |
| 새 실패 케이스 | - | 3건 | ⚠ 트레이스 확인 필요 |

코드를 짜는 비중보다 **트레이스를 읽고, 회귀 케이스를 분류하고, 평가 셋을 늘리는 일**의 비중이 점점 커진다. 모델 출력을 한 줄씩 따라가며 "여기서 왜 이렇게 판단했나"를 추적하는 게 일상이 된다.

## 마무리

하네스 엔지니어링은 추상적인 개념이 아니라 **레이어로 쌓인 시스템 프롬프트, 컨텍스트의 시간축 운용, 툴/권한/루프의 정책 묶음, 그리고 트레이스 기반 평가 루프**다. 이걸 그림으로 한 번 그려보고 나면, "왜 같은 모델인데 도구마다 결과가 다른가"가 추상이 아닌 구체로 보인다.

다음에 AI 도구를 평가할 때, 모델 이름만 보지 말고 **그 도구가 어떤 하네스를 깔고 있는지**를 같이 보자. 거기에 진짜 차이가 있다.
